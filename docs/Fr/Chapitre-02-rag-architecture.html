Vue d'ensemble : le RAG en deux phases

Le RAG est un pattern architectural qui résout les trois limites des LLMs en introduisant une étape de récupération d'information avant la génération. Son nom complet — Retrieval-Augmented Generation — décrit exactement cette idée : la génération est augmentée par un retrieval préalable.

La grande idée en une phrase

Le RAG sépare deux responsabilités : la base de connaissances stocke les faits précis et actualisés, le LLM comprend les questions et synthétise les réponses. Ils collaborent à chaque requête sans que le modèle n'ait besoin d'être ré-entraîné.

Les deux phases fondamentales

Offline ⚙️ Phase 1 — Indexation (une fois, en amont)

Cette phase prépare la base de connaissances. Elle est réalisée une fois (ou lors de mises à jour). Elle ne fait pas intervenir le LLM directement dans la plupart des cas. Le résultat est une base vectorielle prête à répondre aux requêtes en millisecondes.

📄 Documents bruts → ✂️ Chunking → 🔢 Embedding → 🗄️ Vector DB
Online — temps réel ⚡ Phase 2 — Inférence (à chaque requête)

Cette phase se produit à chaque fois qu'un utilisateur pose une question. Elle doit être rapide (idéalement < 500ms pour le retrieval seul). Le LLM intervient uniquement ici.

❓ Question → 🔢 Embedding → 🔍 Recherche → 📋 Top-K → 📝 Prompt → 🤖 LLM → ✅ Réponse

Le pipeline complet en schéma

03

Phase 1 — Indexation en détail

La phase d'indexation est le travail préparatoire. Elle détermine la qualité de toute la chaîne RAG. Un retrieval médiocre ne peut pas être compensé par un excellent LLM — la qualité de l'indexation est le facteur n°1 de la qualité finale du système.

Garbage in, garbage out

La loi fondamentale du RAG : si votre base vectorielle contient des données mal découpées, mal nettoyées, ou mal indexées, le LLM le plus puissant au monde ne pourra pas produire de bonnes réponses. Investissez dans la qualité de l'indexation avant de choisir votre LLM.

Étape 3 — Le Chunking (Découpage)

Le chunking est l'une des décisions les plus importantes — et les plus sous-estimées — dans un système RAG. Il consiste à découper les documents en morceaux (chunks) de taille appropriée avant de les vectoriser.

Pourquoi ne pas vectoriser le document entier ?

🌊
Dilution du signal
Un vecteur représentant 50 pages ne peut pas capturer précisément le sens d'une seule phrase. Le signal sémantique est noyé.
🎯
Précision du retrieval
On veut retrouver le passage exact pertinent, pas le document entier. Un chunk granulaire donne une meilleure correspondance.
💰
Coût du contexte
Injecter un document entier de 100 pages dans le prompt à chaque requête est prohibitif. Les chunks permettent de sélectionner uniquement ce qui est pertinent.

Démonstration visuelle du chunking

Article 12 - Congés annuels. Tout salarié bénéficie de 25 jours ouvrés de congés payés par an. Après 3 ans d'ancienneté, ce droit est porté à 28 jours. Après 5 ans d'ancienneté, le salarié bénéficie de 30 jours. Les congés non pris au 31 décembre sont reportés sur l'année suivante dans la limite de 10 jours. Toute demande doit être soumise 15 jours à l'avance.
Chunk 1 — Droit de base (congés standard)
Chunk 2 — Droits par ancienneté
Chunk 3 — Report & procédure
Pourquoi ce découpage est-il bon ?

Chaque chunk porte une idée cohérente et autonome. Quand un utilisateur demande "combien de jours après 3 ans ?", le Chunk 2 sera retourné — et uniquement lui. Sans ce découpage, on aurait injecté les 3 paragraphes pour une question qui ne nécessite que le Chunk 2.

Les 5 stratégies de chunking

📏 Stratégie 1 — Fixed-Size Chunking (taille fixe) ▼

Principe : Découper tous les N tokens (ex: 256, 512, 1024), avec ou sans chevauchement (overlap).

Avantages : Simple, rapide, prévisible.

Inconvénients : Ignore la structure logique du texte. Une phrase peut être coupée en plein milieu entre deux chunks.

Texte : "La réunion du 12 mars a décidé | d'augmenter le budget de 20%." ↑ COUPURE BRUTALE ici Chunk 1 : "La réunion du 12 mars a décidé" ← Incomplet ! Chunk 2 : "d'augmenter le budget de 20%." ← Manque le contexte !

Overlap : Pour atténuer ce problème, on ajoute un chevauchement de 10-20% entre chunks. Chunk 1 et Chunk 2 partagent les derniers/premiers 50 tokens.

Paramètres typiques : chunk_size=512, chunk_overlap=50

📝 Stratégie 2 — Sentence-Based Chunking ▼

Principe : Découper par phrases complètes, puis regrouper les phrases jusqu'à atteindre la taille cible.

Avantages : Chunks sémantiquement cohérents. Aucune phrase coupée en milieu.

Inconvénients : Les chunks peuvent varier beaucoup en taille. Certaines phrases très longues peuvent dépasser la taille cible.

Idéal pour : Textes narratifs, articles, rapports bien structurés.

📖 Stratégie 3 — Paragraph / Structure-Based Chunking ▼

Principe : Utiliser la structure logique du document comme guide de découpage : paragraphes, sections, articles, titres H1/H2/H3.

Avantages : Très bonne cohérence sémantique. Respecte l'intention de l'auteur.

Inconvénients : Nécessite que le document soit bien structuré. Les documents mal formattés (anciens PDF, scans) donnent de mauvais résultats.

Idéal pour : Documentation technique, règlements, wikis, contenus HTML avec balises sémantiques.

🧠 Stratégie 4 — Semantic Chunking (le plus avancé) ▼

Principe : Utiliser un modèle d'embedding pour détecter les ruptures sémantiques dans le texte. On calcule la similarité entre phrases consécutives ; quand la similarité chute brusquement, c'est une frontière naturelle entre chunks.

Phrase 1 : "La politique de congés accorde 25 jours par an." Phrase 2 : "Après 3 ans, ce droit monte à 28 jours." ← sim(1,2) = 0.91 → même chunk Phrase 3 : "Les demandes se font en ligne sur l'intranet." ← sim(2,3) = 0.73 → même chunk Phrase 4 : "Le budget R&D pour 2024 est de 5M€." ← sim(3,4) = 0.12 → RUPTURE → nouveau chunk

Avantages : Chunks sémantiquement homogènes et autosuffisants. Meilleure qualité de retrieval.

Inconvénients : Plus lent (nécessite l'embedding de toutes les phrases). Plus complexe à implémenter.

Implémentation : LangChain SemanticChunker, LlamaIndex SemanticSplitterNodeParser.

🔄 Stratégie 5 — Recursive Character Text Splitting ▼

Principe : Approche pragmatique de LangChain : essayer une liste de séparateurs dans l'ordre de priorité (["\n\n", "\n", ". ", " ", ""]) et utiliser le premier qui produit des chunks de la taille souhaitée.

Avantages : Bon compromis entre simplicité et qualité. Respecte les structures naturelles quand elles existent, revient au caractère sinon.

Inconvénients : Moins précis que le semantic chunking sur des textes complexes.

C'est le chunker par défaut de LangChain et souvent le premier à essayer dans un projet.

La règle d'or du chunking

Principe de cohérence sémantique

Un bon chunk répond à cette question : "Si ce chunk était le seul texte retourné au LLM, serait-il suffisant pour répondre à la question ?" Un chunk doit être autosuffisant (contient assez de contexte), granulaire (parle d'un seul concept), et sans ambiguïté référentielle (éviter les pronoms comme "il" ou "ce dernier" sans antécédent dans le même chunk).

Étape 4 — Génération des Embeddings

Chaque chunk est transformé en un vecteur numérique de haute dimension par un modèle d'embedding. Ce vecteur encode le sens sémantique du chunk dans un espace mathématique où la distance entre vecteurs reflète la similarité de sens.

Comment fonctionne un modèle d'embedding ?

Un modèle d'embedding est lui-même un Transformer (souvent de type BERT bidirectionnel). Il prend un texte en entrée et produit un vecteur de taille fixe. La propriété clé est apprise pendant l'entraînement : deux textes au sens similaire produisent des vecteurs proches dans l'espace.

PROPRIÉTÉ FONDAMENTALE DES EMBEDDINGS ══════════════════════════════════════ Phrase A : "consommation électrique élevée en été" → Embedding(A) = [0.23, -0.87, 0.45, 0.12, ..., 0.67] (768 dims) Phrase B : "pics de demande d'énergie en période chaude" → Embedding(B) = [0.25, -0.84, 0.47, 0.09, ..., 0.71] (768 dims) Phrase C : "recette de tarte aux pommes avec cannelle" → Embedding(C) = [0.91, 0.32,-0.74, 0.55, ..., 0.02] (768 dims) Similarité cosinus : sim(A, B) = cos(θ_AB) = 0.97 ← TRÈS proches (même sens) sim(A, C) = cos(θ_AC) = 0.11 ← TRÈS éloignés (sens différent) → Le retrieval retournera B quand on cherche A, mais pas C.

La similarité cosinus : la mesure de distance standard

Similarité cosinus entre deux vecteurs a et b
sim(a, b) = cos(θ) = (a · b) / (‖a‖ × ‖b‖)

• sim = 1.0 → vecteurs identiques (même sens)
• sim = 0.0 → vecteurs orthogonaux (sens non liés)
• sim = -1.0 → vecteurs opposés (sens contraires)
• Normalisé sur la norme → insensible à la longueur du texte

Règle critique : cohérence du modèle d'embedding

Le modèle d'embedding utilisé pour indexer les documents doit être exactement le même que celui utilisé pour vectoriser les questions lors de l'inférence. Mélanger deux modèles différents produit des espaces vectoriels incompatibles — les comparaisons de similarité deviennent absurdes.

Étape 5 — La Base de Données Vectorielle

Une base de données vectorielle (Vector Database) est un système de stockage spécialisé optimisé pour rechercher rapidement des vecteurs par similarité dans un espace de haute dimension. C'est fondamentalement différent d'une base SQL ou NoSQL.

🔢 Base Vectorielle
✓Recherche par similarité sémantique
✓Nearest Neighbor Search (ANN)
✓Scalable à des milliards de vecteurs
✓Filtrage par métadonnées combiné
✗Pas faite pour les requêtes exactes complexes
✗Résultats approximatifs (ANN)
📊 Base SQL/NoSQL
✓Requêtes exactes (WHERE, JOIN)
✓Transactions ACID
✓Relations complexes entre données
✗Incapable de rechercher par similarité sémantique
✗Lente sur des vecteurs de 1024 dimensions

L'algorithme ANN : Approximate Nearest Neighbor

La recherche exacte du vecteur le plus proche dans un espace de 1024 dimensions avec 10 millions de vecteurs nécessiterait de comparer chaque vecteur un par un — trop lent. Les algorithmes ANN utilisent des structures d'index spéciales pour obtenir une recherche très rapide avec une légère perte de précision (<1% en pratique).

🌲 Algorithme HNSW (Hierarchical Navigable Small World) ▼

L'algorithme le plus populaire aujourd'hui. Construit un graphe hiérarchique multi-couches : les couches supérieures permettent des "sauts" rapides vers la zone approximative, les couches inférieures affinent la recherche localement.

Analogie : Comme un réseau routier — autoroutes pour parcourir de grandes distances rapidement, routes locales pour arriver précisément à destination.

Performances : ~10ms pour 10M vecteurs, recall@10 > 99%. Utilisé par ChromaDB, Weaviate, Qdrant.

🔢 Algorithme IVF (Inverted File Index) — FAISS ▼

Partitionne l'espace vectoriel en clusters (via k-means). Pour une requête, on cherche d'abord dans les clusters les plus proches, puis on fait une recherche exacte dans ces clusters.

Analogie : Un bibliothécaire qui identifie d'abord la bonne étagère (cluster), puis cherche le bon livre dans cette étagère.

Avantage : Excellent en mémoire vive, très rapide pour les grandes collections. FAISS de Meta est la référence open source.

Phase 2 — Inférence en détail

La phase d'inférence se déclenche à chaque requête utilisateur. Elle doit combiner vitesse (latence acceptable) et précision (bons documents retournés). Voici le pipeline étape par étape.

1
Réception de la question
User Query
L'utilisateur formule sa question en langage naturel. Ex: "Combien de jours de congés ai-je après 4 ans ?" Cette question brute peut nécessiter une transformation avant le retrieval (voir Étape 2).
2
Transformation de la question (optionnel)
Query Transformation / Expansion
Dans les systèmes avancés, la question est reformulée pour améliorer le retrieval : expansion de synonymes, reformulation en plusieurs sous-questions, ou génération de HyDE (Hypothetical Document Embedding). Dans un RAG naïf, cette étape est omise.
3
Vectorisation de la question
Query Embedding
La question est transformée en vecteur avec le même modèle d'embedding utilisé lors de l'indexation. Ce vecteur sera comparé aux vecteurs de la base.
4
Recherche de similarité
ANN Search
L'algorithme ANN (HNSW, IVF…) compare le vecteur de la question aux vecteurs indexés et retourne les K vecteurs les plus proches (typiquement K=3 à 10). On peut aussi filtrer par métadonnées (ex: documents de 2024 uniquement).
5
Re-ranking (optionnel mais recommandé)
Cross-Encoder Reranking
Les Top-K candidats sont reclassés par un Cross-Encoder plus précis (mais plus lent). Contrairement aux Bi-Encoders utilisés pour le retrieval initial, un Cross-Encoder examine la paire (question, chunk) conjointement pour scorer leur pertinence — qualité bien supérieure.
6
Construction du prompt augmenté
Prompt Engineering
Les chunks récupérés sont injectés dans un prompt structuré avec la question originale. La qualité de ce prompt est déterminante pour la qualité de la réponse finale.
7
Génération par le LLM
Augmented Generation
Le LLM reçoit le prompt augmenté et génère une réponse en s'appuyant sur les documents fournis. Un bon système instruction-tune le LLM pour citer ses sources et dire "je ne sais pas" quand les documents sont insuffisants.

Étape 4 — Le Retrieval en profondeur

Dense Search vs Sparse Search

Il existe deux familles fondamentales de méthodes de retrieval :

Dense Search (Vectoriel)

Utilise les embeddings neuronaux. Représentation dense — chaque dimension porte un sens distribué.

✓Capture la similarité sémantique
✓Robuste aux paraphrases et synonymes
✓"congés" ↔ "vacances" → trouvés
✗Peut rater des termes très spécifiques (codes produit, acronymes rares)
Sparse Search (BM25/TF-IDF)

Utilise la fréquence des mots. Représentation sparse — chaque dimension = un mot du vocabulaire.

✓Excellente pour les correspondances exactes
✓Noms propres, codes, identifiants techniques
✓Rapide, interprétable
✗Ignore le sens : "voiture" ≠ "automobile"

Hybrid Search : le meilleur des deux mondes

En production, le standard est de combiner les deux approches via une technique de fusion — souvent Reciprocal Rank Fusion (RRF) :

HYBRID SEARCH — RECIPROCAL RANK FUSION (RRF) ════════════════════════════════════════════ Question : "politique congés article 12 après 3 ans" DENSE SEARCH (vectoriel) : SPARSE SEARCH (BM25) : Rang 1 : Chunk_002 (score 0.94) Rang 1 : Chunk_002 (score 12.3) Rang 2 : Chunk_001 (score 0.71) Rang 2 : Chunk_005 (score 8.1) Rang 3 : Chunk_007 (score 0.68) Rang 3 : Chunk_001 (score 7.4) Rang 4 : Chunk_005 (score 0.61) Rang 4 : Chunk_007 (score 5.2) RRF SCORE : RRF(d) = Σ 1/(k + rank_i(d)) où k=60 (constante) Chunk_002 : 1/(60+1) + 1/(60+1) = 0.0328 ← Champion des deux Chunk_001 : 1/(60+2) + 1/(60+3) = 0.0318 Chunk_005 : 1/(60+4) + 1/(60+2) = 0.0317 Chunk_007 : 1/(60+3) + 1/(60+4) = 0.0315 → Classement final fusionné : [Chunk_002, Chunk_001, Chunk_005, Chunk_007]
Recommandation de production

Le Hybrid Search surpasse systématiquement le Dense ou le Sparse seuls, de 5 à 15% sur les benchmarks standards (BEIR, MTEB). C'est la configuration recommandée pour tout système RAG en production. Elasticsearch, Qdrant et Weaviate le supportent nativement.

Étape 6 — La Construction du Prompt Augmenté

Le prompt augmenté est la clé de voûte du RAG. C'est lui qui structure la communication entre vos documents récupérés et le LLM. Un mauvais prompt peut gâcher un excellent retrieval.

Anatomie d'un prompt RAG optimal

Les 4 techniques de prompting pour RAG

TechniquePrincipeAvantagesInconvénients
Stuff Mettre tous les docs dans le prompt Simple, 1 appel LLM Limité par la context window
Map-Reduce Résumer chaque doc, puis combiner Scalable, gère beaucoup de docs Plusieurs appels LLM, plus lent
Map-Rerank Réponse par doc, garder la meilleure Bonne précision N appels LLM
Refine Raffiner itérativement avec chaque doc Cohérence sur beaucoup de docs Lent, coûteux
Recommandation pratique

Pour la plupart des cas d'usage, commencez avec la technique Stuff (K=3 à 5 documents) avec un bon system prompt. Elle est simple, rapide et suffisante dans 80% des cas. Passez à Map-Reduce ou Refine uniquement si vous devez traiter de nombreux documents simultanément.

Étape 7 — La Génération et la Réponse Finale

C'est ici que le LLM entre en jeu. Il reçoit le prompt augmenté et doit produire une réponse utile, précise et sourcée. La séparation des rôles est fondamentale :

Ce que le LLM apporte
✓Compréhension du langage naturel
✓Capacité de synthèse et de résumé
✓Génération de texte fluide et naturel
✓Raisonnement sur les documents fournis
✓Capacité à détecter les incohérences
Ce que le RAG apporte au LLM
✓Faits précis et vérifiables
✓Données récentes et actualisées
✓Connaissances du domaine privé
✓Sources citables et auditables
✓Réduction drastique des hallucinations

RAG vs Fine-Tuning : comparaison rigoureuse

L'analogie fondamentale

Fine-Tuning = faire apprendre des faits par cœur à un médecin.
→ Il "sait" mais si les protocoles changent, il faut le re-former. Coûteux. Lent. Peut oublier l'ancien (catastrophic forgetting).

RAG = donner au médecin accès à une bibliothèque médicale actualisée.
→ Il consulte avant chaque diagnostic. La bibliothèque se met à jour indépendamment.

CritèreFine-TuningRAG
Coût initialTrès élevé (GPU, semaines)Modéré (indexation)
Mise à jour des donnéesRe-entraînement completAjout de documents (minutes)
Traçabilité des sources❌ Impossible (boîte noire)✅ Sources citées
Réduction hallucinationsRéduites, non éliminéesTrès fortement réduites
Données privées⚠️ Risque mémorisation✅ Données restent externes
Scalabilité donnéesLimitée par trainingIllimitée (ajout docs)
Adaptation style/ton✅ Excellent❌ Moins efficace
Latence inférenceFaible (pas de retrieval)+50-200ms (retrieval)
Compliance / légalDifficile à auditerChaque réponse traçable

Quand utiliser quoi ?

🎯
Choisir Fine-Tuning quand…
Vous devez adapter le style ou le ton du modèle. Vous entraînez sur une tâche de classification précise. Vos données changent rarement. Budget et temps disponibles.
🔍
Choisir RAG quand…
Vos données changent fréquemment. Vous avez besoin de traçabilité (compliance, légal). Grande base documentaire hétérogène. Budget ou délai limité.
🤝
Combiner les deux
Fine-Tuning pour le style/format + RAG pour les faits. Exemple : fine-tuner pour répondre en JSON structuré, augmenter avec RAG pour les données métier.

Erreurs fréquentes des débutants

❌ Erreur 1 — Chunks trop grands (Semantic Stuffing) ▼

Symptôme : chunk_size=2000 tokens, pas d'overlap. Le retrieval retourne des chunks qui contiennent l'information mais noyée dans beaucoup de bruit.

Cause : L'intuition "plus c'est grand, plus il y a d'info" est fausse en sémantique vectorielle.

Solution : Commencer avec chunk_size=256-512, overlap=50. Mesurer le recall sur un jeu de test et ajuster. La plupart des études montrent que 256-512 tokens est optimal pour la recherche factuelle.

❌ Erreur 2 — Oublier les métadonnées ▼

Symptôme : Les chunks ne portent aucune information de contexte (source, date, section). La réponse du LLM ne cite aucune source.

Cause : On se concentre sur le contenu et oublie que les métadonnées sont tout aussi importantes.

Solution : Chaque chunk doit porter : source (nom du fichier), date de mise à jour, section/titre parent, numéro de page, auteur si pertinent. Ces métadonnées permettent le filtrage ET l'attribution dans la réponse.

❌ Erreur 3 — Utiliser deux modèles d'embedding différents ▼

Symptôme : Le retrieval retourne des documents complètement hors sujet. Taux de recall catastrophique.

Cause : Indexation avec OpenAI text-embedding-ada-002, puis requêtes avec text-embedding-3-small. Les espaces vectoriels sont différents — les comparaisons sont sans sens.

Solution : Documenter le modèle d'embedding dans les métadonnées de la base vectorielle. Versionner le modèle. Si vous changez de modèle, ré-indexer entièrement.

❌ Erreur 4 — Ne pas tester le retrieval indépendamment du LLM ▼

Symptôme : Mauvaises réponses finales. On accuse le LLM. En réalité, le retrieval retourne les mauvais documents.

Cause : Le pipeline RAG est évalué de bout en bout sans diagnostiquer où se situent les erreurs.

Solution : Évaluer le retrieval séparément avec des métriques dédiées : Recall@K, Precision@K, MRR (Mean Reciprocal Rank), NDCG. Un retrieval avec Recall@3 < 0.7 ne peut pas donner de bonnes réponses finales, peu importe le LLM.

❌ Erreur 5 — Ignorer la sécurité et les permissions ▼

Symptôme : Un employé du service commercial peut interroger et obtenir des informations des documents RH confidentiels.

Cause : La base vectorielle contient tous les documents sans distinction de permissions.

Solution : Implémenter un filtrage par permissions au niveau du retrieval. Chaque chunk doit porter des métadonnées de permissions (who_can_access: ["RH", "Direction"]). Le retrieval filtre en fonction de l'identité de l'utilisateur avant de retourner les résultats.

Facile Exercice 1 — Identifier les phases
Pour chacune des opérations suivantes, indiquez si elle appartient à la Phase 1 (Indexation) ou à la Phase 2 (Inférence) :

a) Transformer la question de l'utilisateur en vecteur.
b) Découper un PDF en chunks de 512 tokens.
c) Appeler le LLM avec le prompt augmenté.
d) Stocker les vecteurs dans ChromaDB.
e) Calculer la similarité cosinus entre la requête et les chunks.
f) Nettoyer et normaliser le texte des documents.
a) Phase 2 — L'embedding de la question se fait à l'inférence, à chaque requête.
b) Phase 1 — Le chunking est une opération d'indexation offline.
c) Phase 2 — L'appel au LLM est online, par requête.
d) Phase 1 — Le stockage des vecteurs est offline.
e) Phase 2 — La recherche de similarité est online (ANN search).
f) Phase 1 — Le preprocessing est offline.
Facile Exercice 2 — Concevoir un chunking
Vous devez indexer le texte suivant pour un système RAG de support client :

"Le produit X200 est disponible en rouge, bleu et noir. Son prix est de 149€. La garantie est de 2 ans. Pour toute réclamation, contactez support@example.com. Le délai de livraison est de 3 à 5 jours ouvrés. Les retours sont acceptés dans les 30 jours suivant l'achat avec emballage d'origine."

Proposez un découpage en chunks cohérents et justifiez chaque frontière.
Chunk 1 — Caractéristiques produit : "Le produit X200 est disponible en rouge, bleu et noir. Son prix est de 149€. La garantie est de 2 ans."

Chunk 2 — Support et réclamations : "Pour toute réclamation, contactez support@example.com."

Chunk 3 — Livraison et retours : "Le délai de livraison est de 3 à 5 jours ouvrés. Les retours sont acceptés dans les 30 jours suivant l'achat avec emballage d'origine."

Justification : Chaque chunk traite d'un sujet distinct. Si un client demande "comment retourner mon produit ?", seul Chunk 3 sera retourné, sans le bruit des infos couleur/prix. Chunk 1 répond à "quelle est la garantie ?", Chunk 2 à "qui contacter ?".
Moyen Exercice 3 — Construire un prompt RAG
Un système RAG pour une banque a récupéré les 2 chunks suivants pour la question "Puis-je obtenir un prêt immobilier si j'ai un CDD ?" :

Chunk A : "Les prêts immobiliers sont accordés sous condition de stabilité professionnelle. Un CDI est généralement requis."
Chunk B : "Par exception, des dossiers CDD peuvent être étudiés si l'ancienneté dans le CDD dépasse 12 mois et si un co-emprunteur en CDI est présent."

Rédigez un prompt system + user complet pour ce cas, incluant les instructions, le contexte et la question.
System :
"Tu es un conseiller bancaire assistant. Réponds UNIQUEMENT à partir des documents fournis. Cite chaque source. Si les documents ne suffisent pas à répondre complètement, indique-le explicitement et suggère de contacter un conseiller. Ne fournis pas de conseil financier personnalisé."

User :
"Documents pertinents :
[DOC 1] : Les prêts immobiliers sont accordés sous condition de stabilité professionnelle. Un CDI est généralement requis.
[DOC 2] : Par exception, des dossiers CDD peuvent être étudiés si l'ancienneté dans le CDD dépasse 12 mois et si un co-emprunteur en CDI est présent.

Question : Puis-je obtenir un prêt immobilier si j'ai un CDD ?"

Réponse attendue : "En règle générale, un CDI est requis [DOC 1]. Cependant, une exception existe : votre dossier peut être examiné si votre CDD dure depuis plus de 12 mois et si vous avez un co-emprunteur en CDI [DOC 2]. Pour votre situation spécifique, je vous recommande de contacter un conseiller."
Difficile Exercice 4 — Architecture complète
Vous devez concevoir un système RAG pour une entreprise de e-commerce avec 500 000 produits. Chaque produit a : un titre, une description (200 mots), des specs techniques (tableau), et un historique de 20 avis clients.

a) Quelle stratégie de chunking proposez-vous et pourquoi ? (Réfléchissez : tout mettre dans un chunk ? Séparer ?).
b) Quelle base vectorielle choisissez-vous pour 500K produits × ~5 chunks = 2.5M vecteurs ?
c) Quel modèle d'embedding pour des descriptions techniques et des avis en français ?
d) Comment gérez-vous une question comme "J'ai besoin d'une imprimante laser couleur sous 300€ compatible Mac" ?
a) Chunking par type de contenu :
Chunk_desc = description complète (sémantique riche pour les requêtes naturelles)
Chunk_specs = specs techniques (pour les filtrages précis : "RAM 16GB", "poids <2kg")
Chunk_reviews = agrégation des 20 avis en 2-3 phrases de synthèse (sentiment, points forts/faibles)
Métadonnées : prix, catégorie, marque, disponibilité → pour le filtrage !

b) Base vectorielle : Qdrant ou Milvus pour 2.5M vecteurs. HNSW index. pgvector serait trop lent à cette échelle. Pinecone si on veut zero-ops.

c) Modèle d'embedding : BGE-M3 (multilingue, 1024 dims, SOTA sur MTEB français) ou Cohere embed-multilingual-v3.

d) Gestion de la requête complexe :
→ Filtrage métadonnées : catégorie=imprimantes, prix ≤ 300€, marque compatible Mac
→ Dense search sur la description pour "laser couleur"
→ Hybrid search + re-ranking par Cross-Encoder
→ Top-3 produits injectés dans le prompt
→ LLM compare et recommande avec justification

Quiz d'auto-évaluation

Q1. Dans un pipeline RAG, quelle opération est réalisée lors de la phase d'indexation (offline) ?
A) Transformer la question utilisateur en vecteur
B) Découper les documents en chunks et les vectoriser
C) Appeler le LLM pour générer la réponse
D) Construire le prompt augmenté
La phase d'indexation comprend : chargement, nettoyage, chunking, embedding, et stockage dans la vector DB. Toutes ces opérations sont offline, avant toute requête utilisateur.
Q2. Quel est le principal avantage du Semantic Chunking par rapport au Fixed-Size Chunking ?
A) Il est beaucoup plus rapide
B) Il produit des chunks de taille identique
C) Il détecte les ruptures sémantiques naturelles dans le texte
D) Il consomme moins de mémoire
Le Semantic Chunking utilise les embeddings pour détecter quand le sujet change dans le texte. Il produit des chunks sémantiquement homogènes et autosuffisants, améliorant la qualité du retrieval au prix d'une plus grande complexité.
Q3. Vous indexez des documents avec le modèle "text-embedding-3-large". Lors de l'inférence, vous utilisez "text-embedding-3-small". Quel est le problème ?
A) La recherche sera plus lente
B) Les espaces vectoriels sont incompatibles — le retrieval sera aléatoire
C) Le modèle "small" n'est pas assez précis pour les questions
D) Il n'y a pas de problème, les deux modèles produisent des espaces similaires
Chaque modèle d'embedding produit un espace vectoriel distinct. Deux modèles différents = deux espaces incompatibles. Comparer un vecteur "large" avec un vecteur "small" n'a aucun sens géométrique — le retrieval retourne des documents au hasard.
Q4. Qu'est-ce que le "Lost in the Middle" et comment le RAG en tient-il compte ?
A) Un bug dans les bases vectorielles HNSW
B) La tendance des LLMs à inventer des informations
C) Les LLMs prêtent moins d'attention aux informations au milieu du contexte
D) La perte de données lors de la vectorisation
Liu et al. (2023) ont montré que les LLMs dégradent leurs performances sur les informations placées au milieu d'un long contexte. En RAG, cela implique : placer les documents les plus pertinents au début ou à la fin du prompt, limiter le nombre de documents injectés (K=3-5), et utiliser le re-ranking pour s'assurer que les meilleurs docs sont en première position.
Q5. Quel est l'avantage principal du Hybrid Search (Dense + Sparse) sur la recherche dense seule ?
A) Il est plus simple à implémenter
B) Il combine la compréhension sémantique ET la précision sur les correspondances exactes
C) Il ne nécessite pas de modèle d'embedding
D) Il réduit le coût de la vectorisation
Le Dense Search capture les synonymes et le sens ("congés" ↔ "vacances"), mais peut rater les identifiants exacts (codes produit "REF-A2347"). Le Sparse Search (BM25) est précis sur les correspondances exactes mais ignore le sens. L'hybridation via RRF combine les deux forces et surpasse chaque approche isolément de 5-15% sur les benchmarks.

Résumé du chapitre

🎯 Ce qu'il faut absolument retenir

1
Le RAG est un pipeline en 2 phases : indexation (offline, prépare la base) et inférence (online, à chaque requête).
2
La qualité de l'indexation est le facteur n°1 de la qualité finale. Garbage in = garbage out.
3
Le chunking est critique : un chunk = une idée cohérente, autosuffisant, sans ambiguïté référentielle. Préférer Semantic ou Structure-Based en production.
4
Le modèle d'embedding doit être identique à l'indexation et à l'inférence. BGE-M3 est le SOTA open-source multilingue.
5
Les bases vectorielles (Qdrant, ChromaDB, FAISS, Pinecone) utilisent ANN (HNSW, IVF) pour chercher en millisecondes parmi des millions de vecteurs.
6
Le Hybrid Search (Dense + Sparse BM25, fusionné par RRF) surpasse les approches mono-modalités de 5-15% — c'est le standard de production.
7
Le prompt augmenté doit inclure : system instructions, documents sourcés avec leurs origines, et la question. La technique Stuff est le bon point de départ.
8
RAG ≠ Fine-Tuning : RAG pour les faits dynamiques/privés/traçables, FT pour le style/format/classification. Ils peuvent être complémentaires.
