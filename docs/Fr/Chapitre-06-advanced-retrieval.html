02

Limites du retrieval vectoriel naïf

Le retrieval naïf consiste à vectoriser la question de l'utilisateur et à chercher directement les chunks les plus proches. Simple et rapide, mais il souffre de plusieurs failles structurelles.

Les 5 problèmes du retrieval naïf

🎯
Problème 1 — Asymétrie question/réponse
La question "Quelle est la capitale de la France ?" et sa réponse "Paris est la capitale..." sont dans des espaces vectoriels différents. Une question n'est pas un document.
🔤
Problème 2 — Questions vagues ou sous-spécifiées
"Parle-moi des congés" est trop vague pour un retrieval précis. Les bons chunks (28 jours après 3 ans) peuvent ne pas être retournés.
🧩
Problème 3 — Questions multi-aspects
"Compare les avantages de Redis vs PostgreSQL pour le cache" nécessite des informations sur Redis ET PostgreSQL — difficile à satisfaire avec une seule recherche.
🏗️
Problème 4 — Questions abstraites
"Quelles sont les meilleures pratiques de sécurité ?" est trop abstraite pour matcher des chunks spécifiques sur les pratiques concrètes.
📉
Problème 5 — Documents pertinents mal classés
Le Top-1 n'est pas toujours le plus pertinent. Un Cross-Encoder peut réordonner les candidats avec beaucoup plus de précision.
La famille des techniques avancées

Les techniques de ce chapitre appartiennent à deux familles : (1) les transformations de requête (HyDE, Multi-Query, Step-Back, Decomposition) — qui améliorent la qualité des vecteurs de recherche avant d'interroger la base ; et (2) les améliorations de ranking (Re-ranking, RAG-Fusion, CRAG, Self-RAG) — qui améliorent la sélection des documents après retrieval.

HyDE — Hypothetical Document Embedding
Générer une réponse hypothétique pour mieux chercher (Gao et al., 2022)
03

HyDE

+10-20% Recall +1 appel LLM LangChain HyDERetriever

L'insight fondamental

Le problème de l'asymétrie question/document : un chunk qui dit "Paris est la capitale de la France" aura un vecteur très différent de la question "Quelle est la capitale de la France ?". Les embeddings de documents et de questions vivent dans des espaces légèrement différents.

Solution HyDE : avant de chercher dans la base, demander au LLM de générer une réponse hypothétique — un document fictif qui "ressemble" à ce que serait la bonne réponse. Puis vectoriser ce document hypothétique et chercher les chunks réels similaires à ce document fictif.

❌ Retrieval naïf

Question → vecteur_question → cherche dans la base

Problème : vecteur_question ≠ style des documents

✅ HyDE

Question → LLM génère doc hypothétique → vecteur_doc_hypothétique → cherche dans la base

Avantage : même style que les vrais documents

HYDE — EXEMPLE CONCRET ══════════════════════ QUESTION : "Comment postuler pour les congés payés ?" ÉTAPE 1 — LLM génère un document hypothétique : Prompt : "Écris un passage de règlement intérieur qui répondrait à la question : 'Comment postuler pour les congés payés ?'" Document hypothétique généré : "Pour soumettre une demande de congés payés, le salarié doit se connecter à l'espace RH de l'intranet et remplir le formulaire de demande de congés. La demande doit être effectuée au minimum 15 jours à l'avance. Le responsable hiérarchique reçoit une notification et doit valider sous 5 jours ouvrés." ÉTAPE 2 — Vectoriser ce document hypothétique vecteur_hypothetique = embed("Pour soumettre une demande de congés...") ÉTAPE 3 — Chercher dans la vraie base vectorielle → Trouve le chunk réel : "Les demandes de congés doivent être soumises via l'intranet au moins 15 jours à l'avance..." sim(vecteur_hypothetique, chunk_reel) = 0.95 ← haute similarité ! Car les DEUX textes sont dans le STYLE d'un règlement intérieur.
Python — HyDE avec LangChain
from langchain.retrievers import HyDERetriever
from langchain.prompts import PromptTemplate
from langchain_openai import ChatOpenAI, OpenAIEmbeddings
from langchain_community.vectorstores import Chroma

# Template pour la génération du document hypothétique
hyde_template = """Écris un passage de règlement intérieur d'entreprise
qui répondrait à la question suivante.
Ne mentionner que des faits pertinents.
Longueur : 2-3 phrases.

Question: {question}

Passage :"""

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
embeddings = OpenAIEmbeddings()
vectorstore = Chroma(embedding_function=embeddings)

hyde_retriever = HyDERetriever.from_llm(
    llm=llm,
    retriever=vectorstore.as_retriever(search_kwargs={"k": 5}),
    prompt_template=hyde_template
)

docs = hyde_retriever.invoke("Comment faire une demande de RTT ?")
Risque : le document hypothétique peut contenir des hallucinations

Le document généré par le LLM peut contenir des informations incorrectes. Ce n'est pas grave — on l'utilise uniquement comme vecteur de recherche, pas comme réponse. Mais si le LLM hallucine beaucoup dans le doc hypothétique, cela peut orienter la recherche dans une mauvaise direction. Utiliser température=0 et des prompts très directifs pour minimiser ce risque.

Multi-Query Retrieval

+15-25% Recall N appels LLM + N searches LangChain MultiQueryRetriever

Le problème : les mots ne capturent pas tout

Une question formulée d'une certaine façon peut rater des chunks pertinents formulés différemment. Exemple : "Comment réduire la consommation d'énergie ?" peut rater des chunks sur "économies d'électricité" ou "optimisation énergétique".

MULTI-QUERY — PIPELINE COMPLET ══════════════════════════════ QUESTION ORIGINALE : "Quels sont les avantages du télétravail pour les employés ?" ÉTAPE 1 — LLM génère N reformulations (N=3-5) : Q1 : "Quels sont les bénéfices du travail à domicile pour les salariés ?" Q2 : "En quoi le remote work améliore-t-il la qualité de vie au travail ?" Q3 : "Quels impacts positifs le travail depuis la maison a-t-il sur les collaborateurs ?" Q4 : "Avantages du home office pour les employés" ÉTAPE 2 — Retrieval indépendant pour CHAQUE reformulation : Search(Q1) → [doc3, doc7, doc12, doc18, doc22] Search(Q2) → [doc3, doc9, doc12, doc15, doc19] Search(Q3) → [doc3, doc12, doc17, doc20, doc22] Search(Q4) → [doc3, doc7, doc12, doc14, doc21] ÉTAPE 3 — Fusion et déduplication par RRF : doc3 : Rang 1 dans Q1, Q2, Q3, Q4 → score RRF très élevé doc12 : Rang 3,3,2,3 → score élevé doc7 : Rang 2 dans Q1, Q4 → score modéré doc9 : Rang 2 dans Q2 seulement → score faible → Top-5 final fusionné : [doc3, doc12, doc7, doc22, doc14] AVANTAGE : doc17 (trouvé seulement par Q3) serait ignoré en retrieval naïf mais capturé ici grâce à la reformulation diversifiée.

logging.basicConfig()

RAG-Fusion

+20-30% NDCG N appels LLM + N searches + RRF Custom implementation

RAG-Fusion est une architecture qui combine Multi-Query et Reciprocal Rank Fusion de façon systématique, produisant un meilleur classement des documents que chaque technique seule.

flowchart TD A["❓ Question originale"] --> B["LLM génère\n4 reformulations"] B --> C["Search Q1 → Top-5"] B --> D["Search Q2 → Top-5"] B --> E["Search Q3 → Top-5"] B --> F["Search Q4 → Top-5"] C & D & E & F --> G["Reciprocal Rank Fusion\nRRF score pour chaque doc"] G --> H["Classement fusionné\nTop-K documents"] H --> I["LLM génère\nréponse finale"] style B fill:#1a1e2a,stroke:#10b981,color:#10b981 style G fill:#1a1e2a,stroke:#2dd4bf,color:#2dd4bf style I fill:rgba(16,185,129,.1),stroke:#10b981,color:#e8eaf2
Python — RAG-Fusion avec RRF custom
from langchain_core.output_parsers import StrOutputParser
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI
from collections import defaultdict

# 1. Génération de requêtes alternatives
query_gen_prompt = ChatPromptTemplate.from_messages([("human",
    """Génère 4 reformulations différentes de cette question.
    Retourne SEULEMENT les questions, une par ligne.
    Question: {question}"""
)])
generate_queries = (
    query_gen_prompt
    | ChatOpenAI(temperature=0.3)
    | StrOutputParser()
    | (lambda x: x.split("\n"))
)

# 2. Reciprocal Rank Fusion
def reciprocal_rank_fusion(results_list: list[list], k: int = 60):
    """Fusionne plusieurs listes de documents classés par RRF."""
    fused_scores = defaultdict(float)
    doc_map = {}
    
    for docs in results_list:
        for rank, doc in enumerate(docs):
            doc_id = doc.page_content[:100]  # clé unique
            fused_scores[doc_id] += 1.0 / (k + rank + 1)
            doc_map[doc_id] = doc
    
    # Trier par score décroissant
    sorted_ids = sorted(fused_scores, key=fused_scores.get, reverse=True)
    return [doc_map[doc_id] for doc_id in sorted_ids]

# 3. Pipeline complet
def rag_fusion_retrieval(question: str, vectorstore, top_k: int = 5):
    queries = generate_queries.invoke({"question": question})
    queries.append(question)  # inclure la question originale
    
    all_results = [
        vectorstore.similarity_search(q, k=top_k)
        for q in queries if q.strip()
    ]
    
    fused_docs = reciprocal_rank_fusion(all_results)
    return fused_docs[:top_k]

Step-Back Prompting

+12% sur questions multi-hop +1-2 appels LLM LangChain custom chain

L'analogie : le médecin et le symptôme

Un patient dit "j'ai mal à la tête depuis 3 jours". Un bon médecin ne cherche pas immédiatement "traitement mal de tête 3 jours" — il prend du recul et cherche d'abord le contexte général (pathologies possibles avec ce symptôme, facteurs de risque) avant de diagnostiquer.

STEP-BACK PROMPTING — MÉCANISME ═════════════════════════════════ QUESTION SPÉCIFIQUE (difficile à retriever directement) : "Quelles sont les implications légales pour une PME française qui emploie un salarié en CDD depuis 18 mois sans interruption ?" ÉTAPE 1 — LLM génère la "step-back question" (question abstraite) : "Quelles sont les règles générales sur la durée des CDD et leur requalification en CDI en droit du travail français ?" ÉTAPE 2 — Retrieval sur la question abstraite : → Retourne les chunks sur le droit du travail CDD général ÉTAPE 3 — Retrieval sur la question spécifique : → Retourne les chunks sur les cas pratiques PME/CDD ÉTAPE 4 — LLM génère la réponse avec les deux contextes : Contexte général (step-back) + Contexte spécifique (original) → Réponse plus complète et précise RÉSULTAT : La réponse inclut le cadre légal général (requalification automatique après 18 mois ou 2 renouvellements) ET les implications pratiques spécifiques aux PME.
Python — Step-Back Prompting
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI
from langchain_core.output_parsers import StrOutputParser

# Génération de la step-back question
step_back_prompt = ChatPromptTemplate.from_messages([
    ("system", """Tu es un expert en droit du travail. Pour une question spécifique,
génère une version plus générale et abstraite qui capture le cadre conceptuel.
Cette question abstraite permettra de trouver le contexte général."""),
    ("human", "Question spécifique : {question}\nQuestion abstraite :")
])

llm = ChatOpenAI(model="gpt-4o", temperature=0)
step_back_chain = step_back_prompt | llm | StrOutputParser()

def step_back_retrieval(question, retriever):
    # Générer la question abstraite
    abstract_question = step_back_chain.invoke({"question": question})
    
    # Retrieval sur les deux questions
    specific_docs = retriever.invoke(question)
    abstract_docs = retriever.invoke(abstract_question)
    
    # Combiner (déduplication)
    seen, combined = set(), []
    for doc in abstract_docs + specific_docs:
        k = doc.page_content[:80]
        if k not in seen:
            seen.add(k); combined.append(doc)
    return combined[:6]

Query Decomposition

Essentiel pour questions multi-hop N appels LLM + N searches LangChain LCEL

Le problème des questions multi-hop

Certaines questions nécessitent plusieurs étapes de raisonnement. Exemple : "Quelle est la rémunération maximale pour un salarié au SMIC avec 5 ans d'ancienneté et une prime d'ancienneté de 3% ?" nécessite de connaître le SMIC actuel, la règle d'ancienneté ET le calcul de prime.

QUERY DECOMPOSITION — DEUX STRATÉGIES ═══════════════════════════════════════ STRATÉGIE 1 — LEAST-TO-MOST (séquentielle, chaque réponse alimente la suivante) Q complexe : "Quelles entreprises similaires à TechCorp ont le meilleur track record en RAG deployment ?" SQ1 : "Quelles sont les caractéristiques principales de TechCorp ?" → [Chunk: TechCorp est une ESN spécialisée IA, 500 personnes...] → Réponse1: "ESN IA, 500 employés, spécialisation NLP" SQ2 : "Quelles ESN IA similaires à TechCorp (ESN IA, 500 employes) existent sur le marché ?" → Utilise Réponse1 comme contexte pour affiner SQ2 → [Chunks: DataCorp, AIStudio, NLPTech...] SQ3 : "Parmi DataCorp, AIStudio, NLPTech, lesquels ont déployé RAG ?" STRATÉGIE 2 — PARALLÈLE (sous-questions indépendantes) Q complexe : "Compare les congés et la télétravail dans les accords d'entreprise" SQ1 : "Quelle est la politique de congés ?" → [chunks congés] SQ2 : "Quelle est la politique de télétravail ?" → [chunks téléravail] (exécutées en parallèle, fusionnées ensuite)
Python — Decomposition parallèle
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI
import json, asyncio

decompose_prompt = ChatPromptTemplate.from_messages([("human",
"""Décompose cette question en sous-questions simples et indépendantes.
Retourne un JSON {"sub_questions": ["SQ1", "SQ2", ...]}.
Question: {question}""")])

async def parallel_decomposition_retrieval(question, retriever):
    # Décomposer
    llm = ChatOpenAI(model="gpt-4o-mini", temperature=0,
                     response_format={"type": "json_object"})
    response = await (decompose_prompt | llm).ainvoke({"question": question})
    sub_questions = json.loads(response.content)["sub_questions"]
    
    # Retrieval parallèle (async) pour chaque sous-question
    tasks = [retriever.ainvoke(sq) for sq in sub_questions]
    results = await asyncio.gather(*tasks)
    
    # Fusionner et dédupliquer
    seen, all_docs = set(), []
    for docs in results:
        for doc in docs:
            k = doc.page_content[:80]
            if k not in seen: seen.add(k); all_docs.append(doc)
    
    return all_docs, sub_questions

Re-Ranking avancé en production

Le re-ranking est l'étape qui suit le retrieval vectoriel (Bi-Encoder) : on prend les Top-20 à 50 candidats et on les reclasse avec un modèle plus précis (Cross-Encoder) qui analyse la paire (question, document) conjointement.

Architecture retrieve → rerank → generate

1
Retrieval vectoriel (Bi-Encoder) — rapide
ANN search sur toute la base → Top-50 candidats en ~10ms. Précision ~85%. Le Bi-Encoder a pré-calculé tous les vecteurs documents.
2
Re-ranking (Cross-Encoder) — précis
Score de pertinence pour chaque paire (question, candidat_i) → Reclassement des 50 candidats en ~100-200ms. Précision ~95%.
3
Sélection Top-K
Garder les Top-3 à 5 après re-ranking. Ces candidats ont été doublement filtrés pour leur pertinence.
4
Génération LLM
Prompt augmenté avec les Top-K documents les plus pertinents → réponse de haute qualité.

Python — Re-ranking avec sentence-transformers
from sentence_transformers import CrossEncoder

# Charger le modèle de re-ranking
reranker = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")

def retrieve_and_rerank(question: str, vectorstore, top_k: int = 5):
    # Étape 1 : retrieval large (top_k * 10)
    candidates = vectorstore.similarity_search(question, k=top_k * 10)
    
    # Étape 2 : préparer les paires (question, chunk)
    pairs = [(question, doc.page_content) for doc in candidates]
    
    # Étape 3 : score de pertinence via Cross-Encoder
    scores = reranker.predict(pairs)  # score entre -inf et +inf (logit)
    
    # Étape 4 : trier et retourner les top_k
    ranked = sorted(zip(candidates, scores), key=lambda x: x[1], reverse=True)
    return [doc for doc, score in ranked[:top_k]]
Cohere Rerank — La solution API la plus simple

Cohere Rerank v3 offre un service de re-ranking via API en quelques lignes de code, multilingue (100+ langues), avec d'excellentes performances. Idéal si vous voulez le bénéfice du re-ranking sans héberger un modèle. Tarif : ~$2/1000 docs re-rankés.

CRAG — Corrective RAG

CRAG (Corrective Retrieval-Augmented Generation), Yan et al. 2024, introduit une étape d'évaluation et de correction du retrieval. Au lieu d'injecter aveuglément les documents récupérés dans le prompt, on évalue leur pertinence et on adapte la stratégie en conséquence.

CRAG — PIPELINE AVEC AUTO-CORRECTION ═══════════════════════════════════════ Question → Retrieval initial → [doc1, doc2, doc3, doc4, doc5] │ ┌────────────▼────────────┐ │ ÉVALUATEUR DE PERTINENCE│ │ (LLM ou modèle dédié) │ │ │ │ doc1: PERTINENT ✅ │ │ doc2: PERTINENT ✅ │ │ doc3: AMBIGU ⚠️ │ │ doc4: NON PERTINENT ❌ │ │ doc5: NON PERTINENT ❌ │ └────────────┬────────────┘ │ ┌───────────────────┼───────────────────┐ │ │ │ CORRECT (✅) AMBIGU (⚠️) INCORRECT (❌) (≥1 pertinent) (confiance faible) (0 pertinent) │ │ │ Utiliser Web search Web search directement pour compléter pour remplacer │ │ │ └───────────────────┴───────────────────┘ │ LLM génère réponse avec documents validés
Python — CRAG simplifié avec évaluation LLM
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI
from langchain_core.output_parsers import StrOutputParser
from langchain_community.tools import TavilySearchResults

# Évaluateur de pertinence
relevance_prompt = ChatPromptTemplate.from_messages([("human",
"""Évalue si ce document est pertinent pour répondre à la question.
Réponds uniquement par "oui", "non", ou "partiel".

Question: {question}
Document: {document}

Pertinence:""")])

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
evaluator = relevance_prompt | llm | StrOutputParser()
web_search = TavilySearchResults(k=3)

def corrective_rag(question, retrieved_docs):
    # Évaluer chaque doc
    evaluations = [
        evaluator.invoke({"question": question, "document": doc.page_content})
        for doc in retrieved_docs
    ]
    
    relevant_docs = [d for d, e in zip(retrieved_docs, evaluations) if "oui" in e.lower()]
    partial_docs  = [d for d, e in zip(retrieved_docs, evaluations) if "partiel" in e.lower()]
    
    if len(relevant_docs) >= 2:
        # Suffisamment de docs pertinents
        return relevant_docs
    elif partial_docs:
        # Compléter par web search
        web_results = web_search.invoke(question)
        return partial_docs + web_results
    else:
        # Remplacer par web search
        return web_search.invoke(question)

Self-RAG — RAG qui s'auto-évalue

Self-RAG (Asai et al., 2023) est une architecture où le LLM lui-même décide quand chercher des informations, si ce qu'il a trouvé est pertinent, et si sa réponse générée est supportée par les sources. Il génère des tokens spéciaux de contrôle.

SELF-RAG — TOKENS SPÉCIAUX DE CONTRÔLE ══════════════════════════════════════════ Le modèle génère 4 types de "reflection tokens" : [RETRIEVE] → Décision de chercher ou pas "Est-ce que je dois chercher des informations ?" Oui → déclenche un retrieval Non → répond depuis les paramètres du modèle [ISREL] → Pertinence du document récupéré "Ce document est-il pertinent pour ma question ?" Valeurs : relevant / irrelevant [ISSUP] → Support de la génération par le document "Ce que je génère est-il soutenu par les sources ?" Valeurs : fully supported / partially supported / no support [ISUSE] → Utilité de la réponse générée "Ma réponse est-elle utile pour l'utilisateur ?" Score : 1-5 EXEMPLE DE GÉNÉRATION SELF-RAG : Input : "Qu'est-ce que le diabète de type 2 ?" → [RETRIEVE: oui] → retrieval → doc1 récupéré → [ISREL: relevant] → Génère : "Le diabète de type 2 est caractérisé par..." → [ISSUP: fully supported] ← vérifie que c'est dans doc1 → [ISUSE: 5] ← juge l'utilité
Self-RAG vs CRAG : quelle différence ?

CRAG utilise un évaluateur externe (LLM séparé) pour évaluer les documents récupérés puis décide de compléter ou remplacer le retrieval. Self-RAG fine-tune un seul LLM pour qu'il génère lui-même des tokens de contrôle pendant la génération. Self-RAG est plus élégant (un seul modèle) mais nécessite un fine-tuning dédié. CRAG est plus flexible (aucun fine-tuning requis, plug-and-play).

La stack recommandée pour la production

Stack recommandée en production (par ordre de priorité)

Base obligatoire : Hybrid Search (Dense + Sparse BM25) + Re-ranking Cross-Encoder. Cela couvre 80% des gains possibles.

Si les questions sont variées : Ajouter Multi-Query ou RAG-Fusion.

Si les questions sont complexes : Ajouter Query Decomposition ou Step-Back.

Si le corpus peut être incomplet : Ajouter CRAG avec fallback web search.

Ne pas tout activer en même temps — chaque technique ajoute de la latence. Mesurer l'impact de chaque ajout avec des métriques objectives avant de passer au niveau suivant.

Erreurs fréquentes

❌ Erreur 1 — Activer toutes les techniques sans mesurer l'impact ▼

Multi-Query + HyDE + Step-Back + Decomposition + Re-ranking = 5+ appels LLM par requête, latence de 5-10 secondes, coût x5. Sans mesure de l'impact de chaque technique, on peut dépenser beaucoup pour des gains marginaux. Toujours ajouter une technique à la fois et mesurer le delta de qualité vs coût.

❌ Erreur 2 — Utiliser HyDE sans ajuster le prompt selon le domaine ▼

Le document hypothétique de HyDE doit avoir le même style que les vrais documents. Un prompt générique ("génère un texte sur ce sujet") produit un document dans un style différent de celui des vrais chunks. Toujours spécifier le format attendu dans le prompt HyDE : "Génère un extrait de [type de document : règlement, article scientifique, etc.] qui répondrait à cette question."

❌ Erreur 3 — Re-ranking sur trop peu de candidats initiaux ▼

Si le retrieval initial ne retourne que 5 candidats et que le bon document n'en fait pas partie, le re-ranking ne peut pas le faire apparaître. Le re-ranking réordonne les candidats existants — il ne peut pas créer de nouveaux candidats. Toujours retriever large (Top-20 à 50) avant de re-ranker pour Top-5.

❌ Erreur 4 — Confondre Query Decomposition et Multi-Query ▼

Multi-Query : génère des reformulations de la MÊME question, pour capturer différentes façons d'exprimer le même concept. Toutes les sous-requêtes cherchent le même type d'information.

Query Decomposition : découpe une question en sous-questions portant sur des ASPECTS DIFFÉRENTS, chacune nécessitant une recherche indépendante. Utilisé pour les questions multi-hop ou multi-thèmes.

❌ Erreur 5 — CRAG avec un évaluateur trop conservateur ▼

Si l'évaluateur de CRAG est trop strict (marque "non pertinent" trop souvent), le système déclenchera des web searches inutiles, augmentant la latence et le coût. Calibrer l'évaluateur sur un jeu de test pour trouver le bon seuil de pertinence. En pratique, un léger biais vers "partiellement pertinent" est préférable pour éviter les false negatives.

Exercices

Facile Exercice 1 — Choisir la bonne technique
Pour chacun des problèmes suivants, quelle technique de retrieval avancé recommanderiez-vous ?

a) Une question sur la "dépréciation des actifs" dans une base de documents comptables, mais les documents parlent d'"amortissement". Le retrieval naïf échoue car le vocabulaire diffère.
b) "Explique-moi le fonctionnement complet de la chaîne d'approvisionnement de l'entreprise, en incluant les fournisseurs, la logistique et la distribution." (Question très large avec plusieurs aspects).
c) Le retrieval retourne les 5 documents les plus proches, mais l'évaluation montre que le 4ème est souvent le plus pertinent.
d) Parfois une question tombe hors du domaine de la base documentaire et le LLM répond avec confiance en se basant sur ses paramètres internes (pas de retrieval).
a) Multi-Query — générer des reformulations avec le vocabulaire alternatif ("amortissement", "dépréciation", "dotation aux amortissements") pour capturer tous les chunks pertinents indépendamment de la terminologie utilisée.

b) Query Decomposition — décomposer en SQ1: "Qui sont les fournisseurs ?", SQ2: "Comment fonctionne la logistique ?", SQ3: "Comment fonctionne la distribution ?" — chaque sous-question fait un retrieval indépendant.

c) Re-ranking (Cross-Encoder) — le ranking initial est imparfait. Un Cross-Encoder analyse la paire (question, doc4) avec plus de précision et le remonterait en Top-1 ou 2.

d) CRAG — l'évaluateur détecterait que les documents récupérés ne sont pas pertinents et déclencherait un web search ou signalerait l'absence d'information dans la base, évitant que le LLM réponde par des hallucinations.
Moyen Exercice 2 — Implémenter RAG-Fusion from scratch
Vous avez une base vectorielle et une question : "Quels sont les impacts environnementaux de la production d'énergie solaire ?"

a) Proposez 4 reformulations que le LLM pourrait générer pour cette question.
b) Voici les résultats fictifs pour chaque reformulation :
Q1: [A, B, C, D, E] | Q2: [B, A, F, G, C] | Q3: [A, C, H, B, I] | Q4: [J, A, B, K, C]
Calculez le score RRF (k=60) pour les documents A, B et C.
c) Quel serait le classement final de ces 3 documents ?
a) Reformulations :
Q1 : "Empreinte carbone des panneaux photovoltaïques"
Q2 : "Pollution liée à la fabrication de cellules solaires"
Q3 : "Cycle de vie environnemental des installations photovoltaïques"
Q4 : "Problèmes écologiques de l'énergie solaire"

b) Calcul RRF (k=60) :
RRF(doc) = Σ 1/(60 + rang_dans_Qi)

Doc A : Q1(rang1) + Q2(rang2) + Q3(rang1) + Q4(rang2) = 1/61 + 1/62 + 1/61 + 1/62 = 0.01639 + 0.01613 + 0.01639 + 0.01613 = 0.06504

Doc B : Q1(rang2) + Q2(rang1) + Q3(rang4) + Q4(rang3) = 1/62 + 1/61 + 1/64 + 1/63 = 0.01613 + 0.01639 + 0.01563 + 0.01587 = 0.06402

Doc C : Q1(rang3) + Q2(rang5) + Q3(rang2) + Q4(rang5) = 1/63 + 1/65 + 1/62 + 1/65 = 0.01587 + 0.01538 + 0.01613 + 0.01538 = 0.06276

c) Classement : A (0.065) > B (0.064) > C (0.063). Doc A est champion car il apparaît en rang 1 dans deux requêtes différentes.
Difficile Exercice 3 — Concevoir un pipeline RAG complet
Vous construisez un assistant RAG pour un service juridique d'une grande entreprise. Les contraintes sont :
• SLA : <3 secondes de latence totale
• Questions très variées : simples ("quelle est la durée du préavis ?") et complexes ("compare les avantages du statut cadre vs non-cadre sur 5 critères")
• Corpus de 100 000 documents (codes, contrats, jurisprudence)
• Fiabilité critique : aucune hallucination acceptable

a) Proposez une architecture complète avec les techniques de ce chapitre.
b) Comment gérez-vous le SLA de 3 secondes avec des techniques qui ajoutent de la latence ?
c) Quelle approche utilisez-vous pour détecter quand la base documentaire ne contient pas la réponse ?
a) Architecture :
Détection du type de question (classifieur léger) → routing vers deux pipelines :
Pipeline Simple (questions factuelles courtes) : Hybrid Search → Re-ranking (Top-50 → Top-5) → LLM → Réponse
Pipeline Complexe (questions multi-aspects) : Query Decomposition (async, en parallèle) → Hybrid Search pour chaque SQ → RRF → Re-ranking → LLM → Réponse

b) Gestion du SLA :
• Classifier en <100ms (petit modèle ou règles heuristiques sur longueur/complexité)
• Pipeline Simple : Hybrid Search (50ms) + Re-ranking CPU (150ms) + LLM (1.5s) ≈ 1.8s ✅
• Pipeline Complexe : Decomposition async (500ms LLM) + N searches parallèles (N×50ms → 50ms total async) + Re-ranking (200ms) + LLM (2s) ≈ 2.8s ✅
• Cache des queries fréquentes → <100ms pour les 20% de questions répétées

c) Détection "hors domaine" (style CRAG) :
Après re-ranking, vérifier le score du Top-1 Cross-Encoder. Si score < seuil (ex: <-1.0 en logit), déclencher une alerte "information insuffisante dans la base". Ne jamais laisser le LLM répondre depuis ses paramètres internes sur des questions juridiques critiques. Afficher explicitement : "Cette information n'est pas disponible dans notre base documentaire. Consultez un juriste."

Quiz d'auto-évaluation

Q1. Dans HyDE, quel est le rôle du document hypothétique ?
A) Il est directement présenté à l'utilisateur comme réponse
B) Il sert uniquement de vecteur de recherche pour trouver de vrais documents
C) Il remplace la base de données vectorielle
D) Il est utilisé pour fine-tuner le modèle d'embedding
Le document hypothétique de HyDE n'est jamais montré à l'utilisateur. Son seul rôle est d'être vectorisé pour produire un vecteur de recherche dans le même espace sémantique que les vrais documents — résolvant l'asymétrie question/document.
Q2. Quelle est la différence fondamentale entre Multi-Query et Query Decomposition ?
A) Multi-Query reformule la même question ; Decomposition cherche des informations différentes
B) Multi-Query est plus lent que Decomposition
C) Multi-Query utilise un Cross-Encoder ; Decomposition utilise un Bi-Encoder
D) Multi-Query ne fonctionne qu'en anglais
Multi-Query génère des paraphrases de la même question (même information cherchée, vocabulaire différent). Query Decomposition génère des sous-questions portant sur des aspects distincts (informations différentes nécessaires). Multi-Query = diversité de formulation ; Decomposition = division du problème.
Q3. Dans RAG-Fusion, pourquoi utilise-t-on RRF plutôt que de simplement additionner les scores de similarité ?
A) RRF est plus rapide à calculer
B) RRF normalise les rangs entre requêtes, évitant les biais dus aux différentes échelles de scores
C) RRF élimine les doublons automatiquement
D) RRF est plus précis mathématiquement
Les scores de similarité cosinus de deux requêtes différentes ne sont pas comparables directement — une requête peut avoir des scores entre [0.7, 0.9] et une autre entre [0.4, 0.6]. RRF utilise uniquement le rang (1er, 2ème, 3ème…) qui est comparable entre requêtes, via la formule 1/(k + rang).
Q4. CRAG introduit quelle étape absente du RAG classique ?
A) La génération de questions alternatives
B) Une évaluation de pertinence des documents récupérés avec correction éventuelle
C) L'utilisation d'un Cross-Encoder pour le retrieval
D) Le fine-tuning du LLM sur les données d'entreprise
CRAG ajoute une étape d'évaluation entre le retrieval et la génération : un évaluateur (LLM ou modèle dédié) juge si les documents récupérés sont pertinents. Si insuffisamment pertinents, il déclenche une correction (web search, reformulation) avant de passer à la génération.
Q5. Step-Back Prompting est particulièrement efficace pour quel type de questions ?
A) Les questions très courtes et précises (une date, un nom)
B) Les questions en langues étrangères
C) Les questions spécifiques qui nécessitent d'abord le contexte général pour bien comprendre
D) Les questions sur des images ou des données visuelles
Step-Back est efficace quand la bonne réponse dépend d'un cadre conceptuel général qui n'est pas explicitement dans la question. Ex: une question sur "les implications légales du CDD de 18 mois" nécessite de récupérer d'abord les règles générales sur les CDD avant de traiter le cas spécifique.

Résumé du chapitre

🎯 Ce qu'il faut absolument retenir

1
Le retrieval naïf souffre de 5 problèmes : asymétrie question/doc, vocabulaire limité, questions vagues, multi-aspects, et mauvais classement initial.
2
HyDE résout l'asymétrie en générant un document hypothétique dans le style des vrais documents, puis en cherchant des documents similaires à ce doc fictif.
3
Multi-Query génère N reformulations de la même question pour capturer différentes expressions d'un même concept, fusionnées par RRF.
4
RAG-Fusion = Multi-Query + RRF systématique = le meilleur default pour la plupart des cas de production (+20-30% NDCG).

6
Query Decomposition découpe les questions multi-hop en sous-questions indépendantes exécutées en parallèle.
7
Le Re-ranking Cross-Encoder (retrieve Top-50 → rerank → Top-5) est le gain de qualité le plus consistant avec le meilleur ROI.
8
CRAG évalue la pertinence des docs récupérés et corrige en déclenchant un web search si nécessaire. Self-RAG intègre cette logique dans le LLM via fine-tuning.
