02

Pourquoi le chunking est le facteur critique n°1

Dans la communauté RAG, il existe une tendance naturelle à se concentrer sur le modèle LLM ("quel GPT utiliser ?") ou le modèle d'embedding ("BGE-M3 ou text-embedding-3 ?"). Ces choix sont importants, mais ils sont tous conditionnés par la qualité du chunking.

La loi fondamentale du RAG

Un mauvais chunking ne peut pas être compensé par un meilleur LLM, un meilleur modèle d'embedding, ou une meilleure base vectorielle. Si le retrieval retourne les mauvais chunks, la chaîne entière échoue — indépendamment de la qualité des autres composants. Garbage chunking → garbage retrieval → garbage answer.

Les deux tensions fondamentales du chunking

Pourquoi "ça dépend" est une vraie réponse

Il n'existe pas de taille de chunk universellement optimale. La stratégie idéale dépend de :

📄
Le type de document
Articles journalistiques vs documentation technique vs transcriptions audio vs données tabulaires — chacun a une structure différente.
❓
Le type de questions
Questions factuelles précises (date, chiffre) vs questions analytiques (comparer, expliquer) ne nécessitent pas la même granularité.
🔢
Le modèle d'embedding
Un modèle limité à 256 tokens ne peut pas traiter des chunks plus longs sans troncature silencieuse.
💰
Les contraintes de contexte
La fenêtre de contexte du LLM et le budget API limitent la taille et le nombre de chunks injectés.

Rappel des stratégies fondamentales

📏
Fixed-Size Chunking
Complexité : Faible Qualité : Basique
Découpage tous les N tokens, avec overlap. Simple mais ignore la structure logique du texte. Peut couper des phrases, des listes, des paragraphes en plein milieu. À utiliser uniquement pour le prototypage rapide ou les textes très uniformes (ex: flux de données IoT).
Python — LangChain RecursiveCharacterTextSplitter
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=512,      # tokens cibles par chunk
    chunk_overlap=64,    # overlap pour éviter les coupures brutales
    separators=["\n\n", "\n", ". ", " ", ""]
)
chunks = splitter.split_text(document_text)
📝
Sentence / Paragraph Chunking
Complexité : Faible Qualité : Bonne
Découpages par phrases complètes (via spaCy ou NLTK) ou par paragraphes naturels (séparateurs \n\n). Meilleure cohérence sémantique que le Fixed-Size. À préférer au Fixed-Size dès que le texte a une structure naturelle (ce qui est presque toujours le cas).
🏗️
Structure-Based Chunking
Complexité : Modérée Qualité : Très bonne
Découpage par éléments structurels du document : H1, H2, H3 en HTML/Markdown ; articles et paragraphes dans les règlements ; fonctions et classes dans le code. La bibliothèque unstructured détecte automatiquement ces éléments dans les PDF, Word et HTML.
Python — Unstructured pour chunking structuré

    elements,

# Chaque chunk inclut le titre de section comme métadonnée
for chunk in chunks:
    print(chunk.metadata.section, chunk.text[:100])

Semantic Chunking — Le plus intelligent

Le Semantic Chunking est la stratégie la plus sophistiquée de la famille "basée sur le sens". Au lieu de se fier à la structure du texte, il utilise les embeddings pour détecter quand le sens change significativement entre phrases consécutives.

Le principe : détecter les ruptures sémantiques

ALGORITHME DE SEMANTIC CHUNKING ═════════════════════════════════ ÉTAPE 1 — Découper en phrases "Article 12 - Congés annuels." → phrase 1 "Tout salarié bénéficie de 25 jours." → phrase 2 "Après 3 ans, ce droit est porté à 28 jours." → phrase 3 "La demande se fait sur l'intranet." → phrase 4 "Le budget marketing est de 5M€." → phrase 5 ← RUPTURE ! "Les dépenses sont contrôlées mensuellement." → phrase 6 ÉTAPE 2 — Calculer la similarité entre phrases consécutives Embedding(phrase_1) [768D] Embedding(phrase_2) [768D] sim(1,2) = 0.91 ← même sujet (congés) sim(2,3) = 0.88 ← même sujet (ancienneté/congés) sim(3,4) = 0.76 ← lié (procédure congés) sim(4,5) = 0.12 ← RUPTURE SÉMANTIQUE MAJEURE ! sim(5,6) = 0.87 ← même sujet (budget) ÉTAPE 3 — Détecter les seuils de rupture Seuil θ = percentile_95(scores) = 0.40 Quand sim(i, i+1) < θ → nouvelle frontière de chunk ÉTAPE 4 — Créer les chunks Chunk A = [phrase_1, phrase_2, phrase_3, phrase_4] (Congés) Chunk B = [phrase_5, phrase_6] (Budget)

Visualisation des scores de similarité entre phrases consécutives — les creux indiquent les frontières de chunks

# "percentile" — seuil basé sur le percentile des distances

Variante avancée : fenêtre glissante (Rolling Window)

Au lieu de comparer seulement la phrase i à la phrase i+1, on compare la phrase i à une fenêtre moyenne des phrases autour d'elle. Cela rend la détection plus robuste aux petites variations de style.

Python — Semantic Chunking avec fenêtre glissante (custom)
import numpy as np
from sklearn.metrics.pairwise import cosine_similarity

def semantic_chunking_with_window(sentences, embedder, window=2, threshold_pct=90):
    """Semantic chunking avec fenêtre glissante pour plus de robustesse."""
    embeddings = embedder.embed_documents(sentences)
    embeddings = np.array(embeddings)
    
    # Calculer la similarité entre groupes de phrases voisins
    similarities = []
    for i in range(window, len(sentences) - window):
        left = np.mean(embeddings[i-window:i], axis=0)
        right = np.mean(embeddings[i:i+window], axis=0)
        sim = cosine_similarity([left], [right])[0][0]
        similarities.append((1 - sim))  # distance (pas similarité)
    
    # Threshold adaptatif au percentile
    threshold = np.percentile(similarities, threshold_pct)
    
    # Créer les chunks aux ruptures
    chunks, current_chunk = [], [sentences[0]]
    for i, dist in enumerate(similarities):
        idx = i + window
        if dist > threshold:
            chunks.append(" ".join(current_chunk))
            current_chunk = [sentences[idx]]
        else:
            current_chunk.append(sentences[idx])
    chunks.append(" ".join(current_chunk))
    return chunks
Coût vs bénéfice du Semantic Chunking

Le Semantic Chunking nécessite d'encoder toutes les phrases deux fois — une fois pour la détection des ruptures, une fois pour l'indexation finale. C'est deux fois plus coûteux en appels d'embedding. Ce coût est acceptable pour des corpus de taille modérée (jusqu'à quelques millions de documents), mais peut devenir significatif à très grande échelle. Dans ce cas, un hybride Semantic + Structure-Based peut être préférable.

Parent-Document Retrieval — La solution aux deux tensions

Le Parent-Document Retrieval est l'une des innovations les plus importantes dans le design des systèmes RAG. Il résout élégamment le dilemme précision vs contexte en séparant ce qu'on indexe de ce qu'on retourne.

L'insight clé

Chercher petit, retourner grand

On indexe des petits chunks (haute précision vectorielle) pour le retrieval. Mais quand un chunk est trouvé, on retourne son document parent (grand chunk avec contexte complet) au LLM. Ainsi, la recherche est précise ET le LLM reçoit assez de contexte.

Architecture Parent-Document

🗂️
Documents Parents (grands) — stockés séparément
Parent_001 : "Article 12 — Congés annuels. Tout salarié bénéficie de 25 jours ouvrés de congés payés par an. Après 3 ans d'ancienneté, ce droit est porté à 28 jours. Après 5 ans, 30 jours. Les congés non pris au 31 décembre sont reportés dans la limite de 10 jours. Les demandes se font sur l'intranet au moins 15 jours à l'avance..." (500 tokens)
📎
Chunks Enfants (petits) — indexés dans la Vector DB
Child_001a (parent_id: 001) :
"Tout salarié bénéficie de 25 jours ouvrés de congés payés par an." (~15 tokens)
Child_001b (parent_id: 001) :
"Après 3 ans d'ancienneté, ce droit est porté à 28 jours." (~14 tokens)
Child_001c (parent_id: 001) :
"Les demandes se font sur l'intranet au moins 15 jours à l'avance." (~14 tokens)

Implémentation avec LangChain

Python — ParentDocumentRetriever LangChain
from langchain.retrievers import ParentDocumentRetriever
from langchain.storage import InMemoryStore
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_community.vectorstores import Chroma
from langchain_openai import OpenAIEmbeddings

# Deux niveaux de splitting
parent_splitter = RecursiveCharacterTextSplitter(chunk_size=1000)  # grands chunks
child_splitter  = RecursiveCharacterTextSplitter(chunk_size=200)   # petits chunks

vectorstore = Chroma(embedding_function=OpenAIEmbeddings())
docstore = InMemoryStore()  # stockage des parents (ex: Redis en prod)

retriever = ParentDocumentRetriever(
    vectorstore=vectorstore,
    docstore=docstore,
    child_splitter=child_splitter,
    parent_splitter=parent_splitter
)

# Indexation : les enfants vont dans la vectorstore,
# les parents vont dans le docstore
retriever.add_documents(documents)

# Retrieval : cherche les enfants, retourne les parents
relevant_docs = retriever.get_relevant_documents(
    "Combien de jours après 3 ans d'ancienneté ?"
)
print(f"Docs retournés: {len(relevant_docs)}, taille: {len(relevant_docs[0].page_content)} chars")

Hierarchical Chunking — Plusieurs niveaux de granularité

Le Hierarchical Chunking généralise le Parent-Document Retrieval en créant une hiérarchie à plusieurs niveaux correspondant à la structure naturelle du document : document entier → sections → sous-sections → paragraphes → phrases.

Structure hiérarchique type

HIERARCHICAL CHUNKING — EXEMPLE : DOCUMENTATION MÉDICALE ═════════════════════════════════════════════════════════ Niveau 0 — Document complet "Protocole de traitement du diabète de type 2 — CHU Lyon 2024" (~5000 tokens) Niveau 1 — Sections principales Section A : "1. Diagnostic et Bilan Initial" (~800 tokens) Section B : "2. Traitement Médicamenteux" (~1200 tokens) Section C : "3. Surveillance et Suivi" (~900 tokens) Niveau 2 — Sous-sections B.1 : "2.1 Première Ligne — Metformine" (~300 tokens) B.2 : "2.2 Bithérapie" (~250 tokens) B.3 : "2.3 Insulinothérapie" (~350 tokens) Niveau 3 — Paragraphes (indexés en vectorDB) B.1.a : "La metformine est le traitement de 1ère ligne..." (~80 tokens) B.1.b : "Contre-indications : IRC sévère (DFG < 30)..." (~60 tokens) B.1.c : "Posologie initiale : 500mg par jour..." (~50 tokens) STRATÉGIE DE RETRIEVAL : Question : "Quelle est la posologie initiale de la metformine ?" → Vector search sur Niveau 3 → trouve B.1.c (PRÉCIS) → Enrichit avec contexte B.1 (sous-section, ~300 tokens) → LLM Question : "Quels sont les traitements du diabète ?" → Vector search sur Niveau 1 → trouve Section B (GLOBAL) → Retourne Section B directement → LLM
Quand utiliser Hierarchical Chunking ?

Idéal pour les documents longs et bien structurés (règlements, manuels techniques, documentation médicale, contrats complexes). Permet une adaptabilité selon la granularité de la question. Coût : complexité d'implémentation et de maintenance de l'index hiérarchique.

Late Chunking — L'embedding contextuel

Le Late Chunking (Günther et al., 2024) est une innovation récente qui résout un problème fondamental : quand on encode un chunk isolément, il perd le contexte du document environnant. Un chunk commençant par "il" ou "ce dernier" ne peut pas être décodé sans le contexte qui précède.

Le problème que Late Chunking résout

LE PROBLÈME DE LA PERTE DE CONTEXTE ══════════════════════════════════════ Document : "La BCE a annoncé une hausse des taux de 0.5%. Cette décision a provoqué une chute des marchés. Elle était attendue par les économistes depuis janvier." Chunking classique → 3 chunks : Chunk 1 : "La BCE a annoncé une hausse des taux de 0.5%." Chunk 2 : "Cette décision a provoqué une chute des marchés." Chunk 3 : "Elle était attendue par les économistes depuis janvier." Problème : Chunk 2 contient "Cette décision" → lequel ? ← AMBIGU seul ! Chunk 3 contient "Elle" → qui ? ← IMPOSSIBLE à résoudre ! Sans contexte, ces chunks sont désambiguïsables par l'embedding uniquement si le modèle peut les mettre en contexte.

La solution Late Chunking

LATE CHUNKING — COMMENT ÇA MARCHE ══════════════════════════════════ ÉTAPE 1 — Encoder le DOCUMENT ENTIER (pas les chunks !) Input : "La BCE a annoncé... Cette décision... Elle était attendue..." Output : [token_embedding_1, token_embedding_2, ..., token_embedding_N] → Chaque token est encodé AVEC le contexte complet du document → "Cette décision" reçoit un embedding qui "sait" qu'il réfère à la BCE ÉTAPE 2 — Chunker APRÈS l'encoding (d'où "Late") Les token embeddings sont agrégés (mean pooling) par groupe correspondant aux frontières de chunks désirées : Chunk_1_embedding = mean(token_emb[0:12]) ← "La BCE... 0.5%." Chunk_2_embedding = mean(token_emb[12:25]) ← "Cette décision... marchés." Chunk_3_embedding = mean(token_emb[25:38]) ← "Elle était attendue..." RÉSULTAT : Les embeddings des chunks INCLUENT le contexte du document ! Chunk_2 "comprend" désormais qu'il s'agit de la décision BCE. Chunk_3 "sait" que "elle" réfère à la décision annoncée. COMPARAISON : Classique : Chunk_2 (isolé) ≈ decision générale quelconque Late : Chunk_2 (contextualisé) ≈ décision BCE sur les taux
Contrainte majeure du Late Chunking

Le Late Chunking nécessite de passer le document entier dans le modèle d'embedding d'un seul coup. Cela implique que le document doit tenir dans la fenêtre de contexte du modèle d'embedding (8192 tokens pour BGE-M3, 32768 pour E5-Mistral). Les documents plus longs doivent d'abord être découpés en sections avant d'appliquer Late Chunking section par section.

Python — Late Chunking avec Jina Embeddings v3
from transformers import AutoModel, AutoTokenizer
import torch
import numpy as np

model_name = "jinaai/jina-embeddings-v3"  # supporte le late chunking
tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModel.from_pretrained(model_name, trust_remote_code=True)

def late_chunking(document, chunk_boundaries):
    """
    document : texte complet
    chunk_boundaries : liste de (start_char, end_char)
    """
    # Tokeniser le document entier
    inputs = tokenizer(document, return_tensors="pt", return_offsets_mapping=True)
    offset_mapping = inputs.pop("offset_mapping")
    
    # Forward pass : obtenir les token embeddings contextualisés
    with torch.no_grad():
        outputs = model(**inputs)
    token_embeddings = outputs.last_hidden_state[0]  # (seq_len, hidden_dim)
    
    # Mapper les token embeddings sur les chunks
    chunk_embeddings = []
    for start_char, end_char in chunk_boundaries:
        # Trouver les indices de tokens correspondant à ce chunk
        token_mask = [
            start_char <= s and e <= end_char
            for s, e in offset_mapping[0].tolist()
        ]
        chunk_token_embs = token_embeddings[torch.tensor(token_mask)]
        # Mean pooling sur les tokens du chunk
        chunk_emb = chunk_token_embs.mean(dim=0).numpy()
        chunk_embeddings.append(chunk_emb)
    
    return np.array(chunk_embeddings)

Proposition Chunking — Le niveau atomique

Le Proposition Chunking (Chen et al., 2023) pousse la granularité à son maximum : chaque chunk est une proposition factuelle atomique — une seule assertion, autosuffisante, sans ambiguïté référentielle.

Qu'est-ce qu'une proposition ?

PROPOSITION CHUNKING — TRANSFORMATION ══════════════════════════════════════ TEXTE SOURCE : "Marie Curie (1867-1934) était une physicienne et chimiste polonaise naturalisée française. Elle fut la première femme à obtenir un prix Nobel (physique, 1903, partagé avec Pierre Curie et Henri Becquerel pour leurs recherches sur les rayonnements). Elle obtint un second prix Nobel en 1911 (chimie) pour la découverte du polonium et du radium." DÉCOUPAGE EN PROPOSITIONS ATOMIQUES : P1 : "Marie Curie était une physicienne et chimiste." P2 : "Marie Curie était d'origine polonaise." P3 : "Marie Curie fut naturalisée française." P4 : "Marie Curie naquit en 1867." P5 : "Marie Curie décéda en 1934." P6 : "Marie Curie fut la première femme à obtenir un prix Nobel." P7 : "Marie Curie reçut le prix Nobel de physique en 1903." P8 : "Le prix Nobel de physique de 1903 fut partagé avec Pierre Curie." P9 : "Le prix Nobel de physique de 1903 fut partagé avec Henri Becquerel." P10: "Ce prix récompensait leurs recherches sur les rayonnements." P11: "Marie Curie reçut le prix Nobel de chimie en 1911." P12: "Marie Curie découvrit le polonium." P13: "Marie Curie découvrit le radium." AVANTAGES : → Question "Quelle année Marie Curie obtint-elle son 2e Nobel ?" → P11 correspondance parfaite (sim ~0.98) → Aucun "bruit" d'autres informations → Question "Quels éléments chimiques Marie Curie a-t-elle découverts ?" → P12 et P13 retournés → Réponse complète et précise

Générer des propositions avec un LLM

Python — Génération de propositions atomiques via LLM
from openai import OpenAI

client = OpenAI()

def extract_propositions(text: str) -> list[str]:
    prompt = """Décompose le texte suivant en propositions atomiques.
Chaque proposition doit :
- Être une seule assertion factuelle complète
- Être autosuffisante (pas de pronoms ambigus)
- Remplacer les pronoms par leurs référents explicites
- Être la plus courte possible tout en restant complète

Retourne une liste JSON de chaînes.

Texte :
{text}

Propositions (JSON uniquement) :"""

    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": prompt.format(text=text)}],
        response_format={"type": "json_object"}
    )
    import json
    result = json.loads(response.choices[0].message.content)
    return result.get("propositions", [])
Coût élevé du Proposition Chunking

Générer des propositions via LLM coûte cher : 1 page de texte (~500 tokens) peut générer 20-30 propositions, nécessitant ~1000 tokens en input LLM. Pour un corpus de 10 000 pages, le coût d'indexation peut dépasser plusieurs centaines de dollars. Réservé aux cas où la précision est critique (médecine, droit) et où le corpus est stable.

Stratégies par type de document

Chaque type de document a ses défis propres

📄 PDF (surtout les documents scannés ou multi-colonnes) ▼

Défis : L'extraction du texte brut d'un PDF multi-colonnes peut mélanger les colonnes et produire du texte incohérent. Les PDF scannés nécessitent OCR. Les tableaux et figures sont mal extraits.

Stratégie recommandée : Utiliser unstructured avec strategy="hi_res" pour une extraction de qualité (utilise un modèle de détection de layout). Combiner avec chunk_by_title pour respecter la structure.

Python — PDF haute qualité avec unstructured
from unstructured.partition.pdf import partition_pdf

elements = partition_pdf(
    filename="rapport.pdf",
    strategy="hi_res",         # détection layout par modèle vision
    infer_table_structure=True, # convertit tableaux en HTML
    extract_images_in_pdf=False
)
💻 Code source (Python, JavaScript, SQL…) ▼

Défis : Le code ne peut pas être découpé comme du texte — couper une fonction en plein milieu la rend inutilisable. La sémantique est différente : une classe est une unité, pas un paragraphe.

Stratégie recommandée : Utiliser des parsers AST (Abstract Syntax Tree) pour découper par unités logiques : fonctions, classes, méthodes. LangChain propose LanguageParser qui gère 15+ langages.

Python — CodeSplitter pour le code source
from langchain.text_splitter import Language, RecursiveCharacterTextSplitter

python_splitter = RecursiveCharacterTextSplitter.from_language(
    language=Language.PYTHON,
    chunk_size=1000,
    chunk_overlap=100
)
# Séparateurs Python dans l'ordre : class, def, if, for, return, etc.
# → Préserve les blocs logiques complets

python_docs = python_splitter.create_documents([python_code])
📊 Données tabulaires (CSV, Excel, bases de données) ▼

Défis : Les données tabulaires ont une structure très différente du texte. Indexer les cellules individuelles est trop granulaire ; indexer les tableaux entiers est trop large.

Stratégie recommandée :

  • Option 1 — Texte généré : Convertir chaque ligne en phrase naturelle ("Le produit X200 coûte 149€, disponible en rouge, bleu, noir.") → très bon pour les requêtes en langage naturel.
  • Option 2 — Métadonnées filtrables : Stocker les colonnes structurées comme métadonnées (catégorie, prix_min, prix_max) et utiliser le filtrage combiné.
  • Option 3 — Text-to-SQL : Pour des requêtes analytiques complexes, convertir la question en SQL via LLM plutôt que faire du RAG classique.
🎙️ Transcriptions audio et dialogues ▼

Défis : Les transcriptions ont peu de ponctuation naturelle et beaucoup de redondance orale. Les changements de locuteur sont des frontières naturelles de sens.

Stratégie recommandée : Utiliser les changements de locuteur (diarization) comme frontières primaires, puis appliquer un Semantic Chunking à l'intérieur de chaque tour de parole pour les transcriptions longues. Inclure le nom du locuteur dans les métadonnées.

📧 Emails et conversations (threading) ▼

Défis : Un fil d'emails inclut les citations imbriquées, ce qui crée du texte dupliqué à chaque réponse. Le contexte d'une réponse dépend des emails précédents.

Stratégie recommandée : Supprimer les citations répétées (ne garder que le message le plus récent de chaque répondant). Indexer le thread complet comme unité avec les métadonnées (expéditeur, date, objet). Ne pas découper en sous-chunks sauf si le thread est très long.

Évaluer et benchmarker sa stratégie de chunking

Comment savoir si votre stratégie de chunking est bonne ? Il faut des métriques objectives, pas seulement une impression qualitative.

Les métriques de qualité du chunking

📊 Cohérence intra-chunk (Intra-Chunk Coherence) ▼

Mesure à quel point les phrases d'un même chunk sont sémantiquement proches entre elles.

Cohérence intra-chunk
ICC = (1/n²) × ΣᵢΣⱼ cos(eᵢ, eⱼ) pour toutes les paires de phrases dans le chunk

Un ICC élevé (proche de 1) indique que le chunk est thématiquement cohérent. Un ICC bas indique qu'il mélange plusieurs sujets → mauvaise frontière de chunk.

🎯 Recall@K sur jeu de test annoté ▼

La métrique la plus directement liée à la performance RAG finale. Créer 50-200 paires (question, chunk_pertinent) manuellement annotées, puis mesurer combien de fois le bon chunk apparaît dans le Top-K retourné.

Recall@K
Recall@K = |{requêtes où le chunk pertinent ∈ Top-K résultats}| / |{total de requêtes}|

C'est le gold standard. Différentes stratégies de chunking produisent souvent des Recall@3 très différents sur le même corpus.

📏 Distribution de la taille des chunks ▼

Analyser la distribution de la taille des chunks produits. Un bon chunking doit :

  • Avoir une distribution relativement concentrée (pas trop de variance)
  • Éviter les chunks trop petits (<50 tokens — trop peu de contenu)
  • Éviter les chunks trop grands (>1000 tokens — risque de dépasser la limite d'embedding)

Recommandation pour démarrer

Parcours recommandé

Étape 1 : Commencer avec RecursiveCharacterTextSplitter (chunk_size=512, overlap=64). Mesurer le Recall@3 sur 50 questions test.
Étape 2 : Si possible, passer à Structure-Based (unstructured chunk_by_title). Remesurer.
Étape 3 : Si le recall est toujours insuffisant, ajouter Parent-Document Retrieval. Simple à implémenter, gain souvent de 10-20%.
Étape 4 : Pour des besoins de précision maximale sur corpus stable, envisager Proposition ou Late Chunking.

Erreurs fréquentes

❌ Erreur 1 — Utiliser les mêmes paramètres pour tous les types de documents ▼

Un seul pipeline de chunking ne convient pas à tous les documents. Du code source, une fiche produit, un article scientifique et un règlement intérieur ont des structures et des besoins radicalement différents. La solution : avoir des pipelines de chunking différenciés selon le type de document, détecté automatiquement ou fourni comme métadonnée.

❌ Erreur 2 — Chunks avec références non résolues ("il", "ce document", "comme mentionné précédemment") ▼

Un chunk autosuffisant ne doit pas contenir de références qui nécessitent le contexte extérieur pour être comprises. Tester chaque chunk avec la question : "Si ce chunk était la seule information disponible, l'utilisateur pourrait-il comprendre la réponse sans contexte supplémentaire ?" Si non, élargir le chunk ou appliquer Late Chunking.

❌ Erreur 3 — Ne pas inclure le titre de section dans le chunk ▼

Un chunk extrait d'une section "Article 12 — Congés" mais qui ne contient pas ce titre devient ambigu s'il parle de "25 jours" — 25 jours de quoi ? Toujours inclure le titre de section parent dans le chunk ou dans ses métadonnées explicites. La technique "chunk-context enrichment" préfixe chaque chunk avec son chemin hiérarchique.

❌ Erreur 4 — Chunks trop petits pour le modèle d'embedding ▼

Des chunks de 20-30 tokens ne fournissent pas assez de contexte pour que le modèle d'embedding les représente correctement. Le vecteur résultant encode peu de sémantique — trop de dimensions sont "nulles" ou bruitées. Minimum recommandé : 50-80 tokens par chunk. En dessous, envisager de regrouper des phrases consécutives.

❌ Erreur 5 — Ne jamais mesurer le recall du retrieval ▼

La plus grande erreur : optimiser le chunking "à l'aveugle", sans mesure objective. Si vous ne mesurez pas le Recall@3 ou le NDCG@5 avec un jeu de test annoté, vous ne pouvez pas savoir si votre chunking est bon ou mauvais. Créer 50 paires de test prend 2-3 heures et peut économiser des semaines de débogage.

Exercices

Facile Exercice 1 — Identifier la stratégie optimale
Pour chacun des corpus suivants, quelle stratégie de chunking recommandez-vous et pourquoi ?

a) 500 articles de presse en français, chacun de 300-800 mots.
b) La documentation technique de Python (15 000 pages HTML).
c) Des transcriptions de 200 réunions d'entreprise (45 min chacune).
d) Un code source Python de 10 000 lignes (plusieurs modules).
a) Articles de presse : Sentence-Based ou Semantic Chunking. Les articles ont peu de structure formelle (pas de titres H2, H3) mais un flux narratif naturel que le Semantic Chunking capturera bien. Taille cible ~250-400 tokens (environ 2-3 paragraphes).

b) Documentation Python : Structure-Based (chunk_by_title) en priorité car la doc Python est très bien structurée (modules, fonctions, paramètres clairement délimités). Combiner avec Parent-Document pour retourner la section entière quand un sous-élément est trouvé.

c) Transcriptions de réunions : Découpage par locuteur (diarization) comme frontières primaires. Pour les longues interventions (>2 minutes de parole), appliquer un Semantic Chunking secondaire. Les métadonnées doivent inclure le nom du locuteur, le timestamp, et le sujet de réunion.

d) Code Python : RecursiveCharacterTextSplitter avec Language.PYTHON (AST-aware). Découper par fonctions et classes comme unités logiques. Inclure la docstring et la signature dans chaque chunk, ainsi que le nom du module parent en métadonnée.
Facile Exercice 2 — Évaluer des chunks manuellement
Analysez les 4 chunks suivants issus d'un règlement intérieur et notez chacun de 1 à 5 (1 = très mauvais, 5 = excellent) en justifiant votre note :

Chunk A : "25 jours. Après 3 ans, ce droit est porté à 28."
Chunk B : "Les congés annuels sont encadrés par la loi du 20 juin 1936. Tout salarié bénéficie de 25 jours ouvrés de congés payés par an."
Chunk C : "Tout salarié bénéficie de 25 jours ouvrés de congés payés par an. Après 3 ans d'ancienneté, ce droit est porté à 28 jours. Après 5 ans, le salarié bénéficie de 30 jours."
Chunk D : "Article 12 — Congés. Tout salarié bénéficie de 25 jours ouvrés de congés payés par an. Après 3 ans d'ancienneté, ce droit est porté à 28 jours. Après 5 ans d'ancienneté, 30 jours. Les congés non pris au 31 décembre sont reportés sur l'année suivante dans la limite de 10 jours. Les demandes doivent être soumises 15 jours à l'avance via l'intranet."
Chunk A : 1/5 — Commence au milieu d'une phrase ("25 jours."), manque complètement de contexte. "25 jours de quoi ?" est impossible à déterminer. Coupure brutale typique du Fixed-Size mal paramétré.

Chunk B : 3/5 — Mélange deux sujets : référence légale historique + règle actuelle. Pas terrible, mais au moins complet. La référence à 1936 peut parasiter le retrieval pour des questions sur le nombre de jours actuels.

Chunk C : 4/5 — Très bon : cohérent (tout sur les durées de congé selon ancienneté), complet, autosuffisant. Seul bémol : manque le titre "Article 12 — Congés" pour contextualiser si le chunk est retourné isolément.

Chunk D : 5/5 — Excellent : inclut le titre de section, couvre un sujet cohérent (la politique de congés complète), autosuffisant, bien structuré. Un utilisateur comprend immédiatement de quoi il s'agit sans contexte externe.
Moyen Exercice 3 — Concevoir un Semantic Chunking
Voici 6 phrases extraites d'un rapport d'entreprise. Les scores de similarité cosinus entre phrases consécutives sont indiqués :

P1 → P2 : 0.88  |  P2 → P3 : 0.91  |  P3 → P4 : 0.14  |  P4 → P5 : 0.85  |  P5 → P6 : 0.79

a) Si le seuil θ est le 80ème percentile des distances (1 - similarité), quelles sont les frontières de chunks ?
b) Quel serait le découpage final ?
c) Pourquoi le score de 0.14 entre P3 et P4 est-il un signal clair de rupture sémantique ?
a) Calcul du seuil :
Distances (1 - similarité) : [0.12, 0.09, 0.86, 0.15, 0.21]
Triées : [0.09, 0.12, 0.15, 0.21, 0.86]
80ème percentile = 0.86 × 0.8 position ≈ ~0.40 (interpolation approximative)
Alternative simple : θ = max(distances) × 0.5 = 0.43
La distance de 0.86 (entre P3 et P4) est clairement la seule à dépasser tout seuil raisonnable.

b) Découpage :
Chunk 1 = [P1, P2, P3] — similarités élevées entre elles
Chunk 2 = [P4, P5, P6] — nouveau sujet

c) Un score de 0.14 indique que P3 et P4 sont presque orthogonaux dans l'espace vectoriel — leurs sujets n'ont presque rien en commun. C'est une rupture sémantique majeure et non ambiguë, bien au-dessus du bruit normal (les autres écarts sont entre 0.09 et 0.21). Dans un rapport d'entreprise, cela correspondrait typiquement à un changement de section (ex: de "résultats financiers" à "stratégie RH").
Difficile Exercice 4 — Architecture de chunking pour corpus hétérogène
Vous concevez le pipeline RAG d'un assistant juridique intelligent pour un cabinet d'avocats. Le corpus comprend :

• 50 000 décisions de justice (PDF scannés, 5-30 pages chacune)
• 200 codes législatifs (HTML bien structuré, articles numérotés)
• 10 000 contrats d'entreprises (Word, structure variable)
• 500 mémos juridiques internes (texte libre, 1-5 pages)

a) Proposez une stratégie de chunking différenciée pour chaque type.
b) Comment structurez-vous les métadonnées pour permettre le filtrage (ex: droit civil vs commercial, date de décision, parties concernées) ?
c) Quelle métrique utiliserez-vous pour valider votre stratégie et comment créez-vous votre jeu de test ?
d) Quel est le risque principal du Proposition Chunking pour les décisions de justice ?
a) Stratégies différenciées :
Décisions de justice (PDF scannés) : unstructured + strategy="hi_res" pour OCR et détection de layout. Structure-Based sur les sections identifiées (attendu, dispositif, motivation). Parent-Document : chunk par paragraphe de raisonnement, parent = décision complète.
Codes législatifs (HTML) : Structure-Based par article (H2/H3). Chaque article = un chunk autonome avec son numéro et le code parent en métadonnée. Hierarchical pour retourner le chapitre complet si besoin.
Contrats (Word) : unstructured chunk_by_title. Overlap important (10%) pour les clauses qui s'enchaînent. Parent = article de contrat, enfant = sous-clause.
Mémos (texte libre) : Semantic Chunking pour détecter les ruptures logiques naturelles.

b) Métadonnées structurées :
{source_type: "decision", domaine: "civil/commercial/penal", juridiction: "TGI Paris", date: "2024-03-15", numero_affaire: "...", parties: ["SAS ABC", "M. Dupont"], matiere: "contrat-bail", confidentialite: "public"}

c) Validation : Constituer 200 questions annotées par des avocats du cabinet (questions factuelles + analytiques). Mesurer Recall@3, Precision@3, MRR. Benchmark comparatif entre les 4 stratégies par type. Seuil acceptable : Recall@3 ≥ 0.85 pour les questions factuelles.

d) Risque du Proposition Chunking pour les décisions : Une décision de justice contient des nuances juridiques critiques liées au contexte (jurisprudence citée, faits de l'espèce). Décomposer en propositions atomiques peut décontextualiser une conclusion juridique de ses conditions d'application. Une proposition "Le locataire doit payer sous 8 jours" sans le contexte "en cas de mise en demeure préalable" peut induire en erreur. Le risque de mauvaise interprétation est réel et potentiellement lourd de conséquences.

Quiz d'auto-évaluation

Q1. Quel est le principal avantage du Parent-Document Retrieval par rapport au chunking simple ?
A) Il réduit le coût des embeddings
B) Il combine la précision de petits chunks pour le retrieval et le contexte de grands chunks pour la génération
C) Il élimine le besoin d'un modèle d'embedding
D) Il permet d'indexer des images
Le Parent-Document Retrieval sépare "ce qu'on indexe" (petits chunks précis, pour un retrieval vectoriel de haute qualité) de "ce qu'on retourne" (le document parent grand, pour donner assez de contexte au LLM). C'est l'une des techniques les plus impactantes pour améliorer la qualité RAG.
Q2. Dans le Semantic Chunking, sur quoi se base-t-on pour détecter les frontières entre chunks ?
A) Le nombre de tokens dans chaque phrase
B) La présence de titrages (H1, H2) dans le texte
C) La chute de la similarité cosinus entre phrases consécutives
D) La présence de pronoms dans le texte
Le Semantic Chunking encode chaque phrase en vecteur et calcule la similarité cosinus entre phrases voisines. Quand cette similarité chute brusquement (dépasse un seuil basé sur le percentile), cela indique un changement de sujet — c'est une frontière naturelle entre chunks.
Q3. Qu'est-ce qui distingue le Late Chunking du chunking classique ?
B) Le Late Chunking encode d'abord le document entier puis agrège les token embeddings contextualisés
A) Le Late Chunking effectue le découpage après l'inférence du LLM
C) Le Late Chunking utilise uniquement des modèles non-neuronaux
D) Le Late Chunking ne nécessite pas de modèle d'embedding
Le Late Chunking inverse l'ordre habituel : au lieu d'encoder des chunks isolément, il encode le document complet en une seule passe pour obtenir des token embeddings contextualisés, puis agrège ces embeddings par fenêtres correspondant aux frontières de chunks. Les embeddings résultants "savent" ce qui précède et ce qui suit chaque chunk.
Q4. Une proposition (Proposition Chunking) est…
A) Un synonyme de chunk de taille fixe
B) Un titre de section de document
C) Une assertion factuelle atomique et autosuffisante, sans pronoms ambigus
D) Une question généré par le LLM sur le document
Une proposition est la plus petite unité d'information qui soit autosuffisante. Elle contient exactement un fait, sans ambiguïté référentielle (pronoms remplacés par leurs référents). C'est la granularité maximale pour un retrieval précis, au prix d'un coût d'indexation élevé (génération via LLM).
Q5. Comment évalue-t-on objectivement la qualité d'une stratégie de chunking ?
A) En comptant le nombre de chunks produits
B) En vérifiant que tous les chunks font la même taille
C) En mesurant le Recall@K sur un jeu de test annoté de paires (question, chunk pertinent)
D) En minimisant le nombre de tokens total indexés
Le Recall@K sur un jeu de test annoté est la métrique gold standard. On crée manuellement des paires (question, chunk pertinent), puis on mesure combien de fois le bon chunk apparaît dans les Top-K résultats de la recherche vectorielle. C'est la seule façon de savoir objectivement si votre chunking améliore le retrieval.
