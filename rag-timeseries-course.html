Part 01

Retrieval-Augmented Generation: Deep Foundations

Why RAG was invented, the problems it solves, and a complete walkthrough of the pipeline from document loading to final answer.

Why RAG Was Invented

💡
Core Tension

LLMs are trained once, but the world changes continuously. They are frozen libraries — brilliant, but unable to know what happened after their graduation day.

To understand why RAG exists, you must first understand the fundamental tension at the heart of modern LLMs. They consume vast knowledge during training, then are sealed. Every enterprise using an LLM faces three converging problems:

Problem 1 — Hallucinations

Technical explanation: LLMs are trained with cross-entropy loss on next-token prediction. The model learns a probability distribution P(token | context). It has no explicit "I don't know" state — it always produces a distribution over the vocabulary. When the conditioning context doesn't contain enough signal, the model falls back on correlational patterns that produce plausible but false completions.

🧠 Analogy

A student who never studied doesn't leave the exam blank — they write something that sounds like an answer, using the subject's structure and vocabulary, but fabricating the details.

Example — Hallucination 
User:  "What was the revenue of Acme Corp in Q3 2024?"
LLM:   "Acme Corp reported revenue of $2.3B in Q3 2024,
        representing 12% YoY growth..."
       → Completely fabricated if Acme Corp isn't in training data

Problem 2 — Knowledge Cutoff

Every LLM has a training cutoff. After that date, the model knows nothing about new research, updated regulations, recent market events, or policy changes. This is catastrophic for enterprises where answers must reflect today's reality.

Problem 3 — Context Window Limitations

Even with 1M-token context windows, a corporate knowledge base might have 10 million documents. Stuffing everything in is not retrieval — it's brute force. It's slow, expensive, and the model's attention diffuses across irrelevant content.

Fine-Tuning vs. RAG

Fine-tuning teaches the model HOW to reason in your domain. RAG gives the model WHAT to reason about. These are complementary, not alternatives.

Architecture

Stage 1
📄 Document Loading
Ingests raw data from heterogeneous sources (PDF, HTML, DOCX, SQL, APIs) into a uniform Document object representation.
IN: PDFs, HTML, DOCX, DBs OUT: List[Document(content, metadata)]
Stage 2
🧹 Cleaning & Preprocessing
Removes noise: HTML tags, headers/footers, watermarks, boilerplate. Improves embedding quality by maximizing signal-to-noise ratio.
IN: Raw Documents OUT: Cleaned Documents
Stage 3 ★
✂️ Chunking
Splits documents into smaller pieces. This is the most impactful design decision in RAG. Strategies: fixed size, recursive character, semantic breakpoint detection.
IN: Clean Documents OUT: Chunks (256–1024 tokens each)
Stage 4
🔢 Embedding Generation
Converts each chunk into a dense numerical vector via a transformer encoder. Similar meaning → geometrically nearby vectors.
IN: Text chunks OUT: Vectors ∈ ℝ⁷⁶⁸
Stage 5
🗄️ Vector Database Storage
Stores (vector, text, metadata) triples in a specialized database with ANN indexing for millisecond-fast similarity search at scale.
IN: (vector, text, metadata) triples OUT: Indexed collection
Stage 6
🔍 Similarity Search
Embeds the user query with the same model, then finds the top-k most similar vectors in the database. Returns ranked chunks.
IN: User query text OUT: Top-k chunks + similarity scores
Stage 7
📝 Prompt Construction
Combines retrieved chunks + user query into a structured prompt. Instructs the LLM to answer using ONLY the provided context — the grounding mechanism.
IN: Top-k chunks + query OUT: System + context + user prompt
Stage 8
🤖 LLM Generation
The LLM synthesizes retrieved context into a coherent, well-written answer. Acts as the reasoning and language layer, not the memory layer.
IN: Constructed prompt OUT: Final answer + source citations

Part 02

Vector Databases

From the mathematics of embedding spaces to production-grade ANN algorithms and database selection.

Why SQL Cannot Do Semantic Search

SQL is designed for exact match and range queries on structured data. Semantic search requires finding documents whose meaning is similar — there is no SQL operator for this. Brute-force comparison at 10M documents with d=768 requires ~7.68 billion multiplications per query.

Similarity Metrics

Math
v₂ / / θ /──────── v₁ COSINE SIMILARITY = (v₁ · v₂) / (|v₁| × |v₂|) Range: -1.0 (opposite) → 0.0 (orthogonal) → +1.0 (identical direction) DOT PRODUCT = Σᵢ v₁ᵢ × v₂ᵢ (= cosine when vectors are unit-normalized) EUCLIDEAN DIST = √(Σᵢ (v₁ᵢ − v₂ᵢ)²) (sensitive to vector magnitude)
⚠️
Why Cosine over Euclidean?

A long document may have a larger-magnitude vector than a short one, even if they discuss the same topic. Cosine similarity normalizes this away — it only cares about direction, not magnitude.

Approximate Nearest Neighbor Algorithms

IVF — Inverted File Index

🧠 Analogy

A city directory. Instead of searching every house, you first identify the neighborhood (cluster centroid), then search within that neighborhood only.

Algorithm
TRAINING PHASE: 1. Run k-means → find k centroids 2. Assign each vector to its nearest centroid 3. Build inverted list: centroid → [vectors in cluster] QUERY PHASE: 1. Find nprobe nearest centroids to query 2. Search only those clusters 3. Return top-k TIME SAVINGS: Search N/k vectors instead of N TRADEOFF: May miss true nearest neighbor in unsearched cluster

HNSW — Hierarchical Navigable Small World ★

🧠 Analogy

A highway system. To drive from one city to another, you take the interstate (Layer 2), not local roads (Layer 0). HNSW builds "express lanes" through vector space.

Structure
Layer 2 (sparse): ●─────────────────●──────● (few long-range connections) Layer 1 (medium): ●──●────●──────●──●────●──● Layer 0 (dense): ●●●●●●●●●●●●●●●●●●●●●●●●●●● (every vector, local connections) SEARCH: 1. Enter at Layer 2 → greedy move toward query 2. Drop to Layer 1 → continue greedy search 3. Drop to Layer 0 → exact neighbors in subgraph Complexity: O(log n) · High recall (>95%) · Dynamic inserts

Architecture

Part 03

Embeddings: From First Principles

How text becomes geometry, why similar meanings produce nearby vectors, and which embedding models to choose.

How Embeddings Are Generated

Pipeline
INPUT TEXT: "Revenue declined sharply in Q3" │ ▼ TOKENIZATION: ["Revenue", "declined", "sharply", "in", "Q3"] → token IDs: [12045, 6823, 9011, 287, 148] │ ▼ TOKEN EMBEDDINGS + POSITIONAL ENCODING: Each token ID → learned vector ∈ ℝ⁷⁶⁸ + sine/cosine positional signal │ ▼ TRANSFORMER ENCODER (L layers): Multi-head self-attention: tokens attend to each other Each token's repr. updated by full context After L layers: contextualized representations │ ▼ POOLING (critical design choice): Option A: [CLS] token ← BERT default Option B: Mean of all tokens ← Sentence-BERT ★ Option C: Max pooling Option D: Weighted mean │ ▼ OUTPUT: Single vector ∈ ℝ⁷⁶⁸ [0.12, -0.87, 0.34, ..., 0.91]

🚨
Domain Mismatch — #1 Embedding Quality Problem

A model trained on Wikipedia and Common Crawl may produce poor embeddings for medical jargon, legal text, or financial formulae. Domain-specific fine-tuning of the embedding model is usually the highest-ROI improvement you can make.

Windowing (The Chunking Analogue)

Sliding Windows
Original: [x₁ x₂ x₃ x₄ x₅ x₆ x₇ x₈ x₉ x₁₀] Window W=4, Stride S=2: Window 1: [x₁ x₂ x₃ x₄] Window 2: [x₃ x₄ x₅ x₆] ← overlap = 2 Window 3: [x₅ x₆ x₇ x₈] Window 4: [x₇ x₈ x₉ x₁₀] Stride=1 → Maximum overlap (expensive) Stride=W → No overlap (may miss cross-boundary patterns) FORECASTING SETUP: Input window: [x₁ ··· x_W] ← look-back history Target: [x_{W+1} ··· x_{W+H}] ← forecast horizon H

Recommendation for RAG

PatchTST for Architecture A: patch-level representations are high-quality and the architecture is well-understood. TS2Vec as alternative: specifically designed for representation learning, handles arbitrary-length windows. Start with PatchTST.

PatchTST — How It Patches Time Series

PatchTST
TIME SERIES: [x₁ x₂ x₃ x₄ x₅ x₆ x₇ x₈ x₉ x₁₀ x₁₁ x₁₂] PATCHES (size=4, stride=2): P₁ = [x₁ x₂ x₃ x₄] P₂ = [x₃ x₄ x₅ x₆] ← overlapping patches P₃ = [x₅ x₆ x₇ x₈] P₄ = [x₉ x₁₀ x₁₁ x₁₂] Each Pᵢ → linear projection → transformer encoder → patch embedding All patch embeddings → mean pool → series embedding ∈ ℝᵈ Analogy: Like ViT image patches, but for 1D numerical sequences. The patch = the "word" of the time series vocabulary.

Part 10

Best Practices

Common mistakes, evaluation metrics, and production considerations for RAG and time series systems.

Common RAG Mistakes

❌ MistakeUsing different embedding models for queries and documents
✓ CorrectQuery and documents MUST use the same embedding model. The vector spaces are model-specific.
❌ MistakeFixed chunk size without evaluation — picking 512 tokens "because it sounds right"
✓ CorrectExperiment with 256, 512, 1024. Evaluate retrieval quality at each. Don't assume.
❌ MistakeEvaluating only the final answer quality, ignoring retrieval quality
✓ CorrectEvaluate retrieval (Precision@k, Recall, MRR) separately. Retrieval failures cascade into generation failures.
❌ MistakeTrusting the LLM to refuse when context doesn't contain the answer
✓ CorrectLLMs still hallucinate even with context. Use faithfulness checks and explicit system prompts.

Common Time Series Mistakes

❌ MistakeTemporal data leakage — retrieving from the future when evaluating on historical data
✓ CorrectEnforce strict temporal train/test splits. Never retrieve windows that occur after the query timestamp.
❌ MistakeUsing Euclidean distance for time series similarity — sensitive to phase shifts
✓ CorrectUse DTW for shape-based matching, or use learned embeddings (PatchTST/TS2Vec) that handle this.
❌ MistakeGlobal normalization only — ignoring per-window local scaling
✓ CorrectConsider local (per-window) z-score normalization for retrieval. Compare both and measure.

Evaluation Metrics

Precision@k
= relevant_retrieved / k
Of the k retrieved chunks, what fraction are actually relevant?
Recall@k
= relevant_retrieved / total_relevant
Of all relevant items, how many did retrieval surface?
MRR
= mean(1 / rank_first_relevant)
How highly ranked is the first relevant result on average?
NDCG@k
= DCG@k / IDCG@k
Quality-weighted ranking metric — penalizes relevant items ranked lower.
Faithfulness
= grounded_claims / total_claims
Fraction of answer claims traceable to retrieved context.
BERTScore
Semantic F1 vs. reference
Semantic similarity between generated answer and gold reference.

Advisor's Final Note

Time series RAG is genuinely novel. The evaluation methodology alone — how do you know if a retrieved window is "truly" relevant? — is a research contribution. Start with Architecture B (simplest), establish baselines, then upgrade to A or C and measure. Don't assume complexity helps: measure everything.
